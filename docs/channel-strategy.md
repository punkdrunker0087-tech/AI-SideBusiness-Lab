# channel-strategy.md — 販売チャネル戦略

このドキュメントは、会社が前提とする販売チャネルとその特性を定義する。
市場調査部・事業評価部は「どの販路で売れる想定か」をこの一覧を基準に
評価し、制作部・マーケティング部は成果物の形式・価格・訴求文をチャネル
特性に合わせて最適化する。

## チャネル一覧

| チャネル | 用途 | 価格帯 | 主な商品 |
|---|---|---|---|
| BOOTH | 個人向け（日本語圏） | ¥100〜¥1,500 | テンプレート・プロンプト |
| note | 記事＋デジタル販売（日本語圏） | ¥100〜¥2,000 | 解説・ノウハウ・テンプレート |
| Gumroad | 海外向け | $5〜$49 | 英語版テンプレート・AIワークフロー |
| GitHub | 集客・OSS | 無料〜スポンサー | サンプル・OSS・ドキュメント |

上記は「販売」チャネル。加えて、直接収益を生まない「プロモーション」
チャネルとしてXを運用する（詳細は次節）。

## プロモーションチャネル: X

X（`docs/brand.md`のアカウント）は販売チャネルではなく、信頼構築と
集客のための発信チャネルとして運用する。

- **投稿頻度ルール（2026-07-09 CEO決定・認知拡大策）**: **1日3本**を
  基準とし、**可能な限り画像付き**で投稿する
  - 内容は商品に限らず、ここまでの経過・気づき・失敗・数字の変化など
    「AIと会社を運営する実験の記録」を中心にする
  - 画像の標準素材: ツールの実画面スクリーンショット、ダッシュボード、
    Before/Afterの図解、リポジトリのコミット画面など「実物」を優先する
  - AIは画像を生成できないため、下書き（`/sns`）に**画像指示を必ず
    添える**。指示は「撮影型」（どの画面をどんな状態で撮るか）または
    「作成型」（構図・文字・ブランド配色までサムネイル指示書と同粒度）
    の2形式。画像内の数字は実績値のみ（架空の数字を入れない）
  - 3本の配分も80:20を維持する（商品紹介はおおむね1日1本まで）
- **投稿時間帯ルール（2026-07-14 CEO決定・実検索の一般知見に基づく）**:
  3本を「見られる時間」に割り当て、`/sns`は各投稿に**推奨投稿時刻**を
  必ず添える。時間を指定しない下書きは不完全とみなす
  - 平日の目安: ①朝7〜9時（通勤層）②昼12〜13時（昼休み）③夜20〜22時
    （最大リーチのゴールデンタイム）。夕方17〜19時も可
  - 土日: 13〜24時、特に18〜22時に寄せる（休日はアクティブ率が高い）
  - **最も届けたい1本（商品紹介・買い手向け）は夜20〜22時**に置く
  - 初動6時間でアルゴリズム評価がほぼ決まるため、深夜すぎる時刻を避ける
  - 最終的には自分のXアナリティクスで自社フォロワーの反応時間を確認し、
    このデフォルトを実データで上書きしていく（当面は上記の一般知見で運用）
- **80:20の法則**を守る。商品の宣伝アカウントにしない
  - 80%: AI活用・Claude Code・業務効率化に関する発信（開発ログ・失敗も含む）
  - 20%: 商品（BOOTH等）の紹介
- **書き出しの原則**: 投稿は必ず「読者の役に立つ情報」から書き始め、
  「LIMIT LABでは〜」のようなブランド主語から書き出さない。ブランド名は
  文末に軽く触れる程度（ハッシュタグ等）に留める。読者が知りたいのは
  「LIMIT LABが何をしたか」ではなく「自分の役に立つ情報」である
- 「LIMIT LAB WORKSはゼロからAI会社OSを実際に作っている」という開発
  記録そのものを一番のコンテンツとして扱う。単なる機能説明ではなく、
  「Claude Codeで"AIがAIに仕事を振る会社"を作っている。今日の改善で
  案件評価の精度が上がった」のように、プロジェクトの背景・面白さを
  伝える投稿を優先する
- GitHubリポジトリも同様に、単なるOSS公開ではなく開発ログ・ストーリーの
  置き場として活用する（外部向けREADMEはこの方針に沿って書く）
- **週次まとめ投稿**: 毎週1本、「今週作ったもの」をまとめて投稿する
  （例: 「Week1 ✔BOOTH公開 ✔GitHub公開 ✔Prompt Pack公開 ✔商品3点
  ✔次はBusiness Template Pack」）。プロジェクトを継続的に追ってくれる
  読者を増やすことが目的

## 使い分けの原則

- **BOOTH・note**: 日本語圏の個人・小規模ビジネス向け。単価は低めだが
  参入障壁が最も低く、最初の実売上を得る検証チャネルとして最優先する
- **Gumroad**: 英語版が用意できた案件で、より高い単価（$5〜$49）を
  狙う。日本語版で実績が出てから横展開するのが基本パターン
- **GitHub**: 直接の販売チャネルではなく、無料サンプル・OSS版の公開に
  よる集客・信頼獲得のチャネルとして使う。有料版（BOOTH/note/Gumroad）
  への導線として位置づける

## 市場調査・事業評価への反映

- 市場調査部は、案件のEvidence収集時に上記チャネルの実売上・価格帯・
  手数料を優先的に調べる（`Research.md`のCompetitors／販売チャネル比較
  セクションで使用する）
- 事業評価部は、案件がどのチャネルで売れる想定かを踏まえて市場性・
  収益性のスコアリングを行う。想定チャネルが上記いずれにも該当しない
  場合は、その旨をEvidence不足として明記する

## マーケティング部への反映

マーケティング部は、この一覧に基づき案件ごとに使用するチャネルを選び、
チャネルごとに公開状態（公開準備中／審査中／公開済み／停止／改善中）を
Portfolio Dashboardおよび `cases/<ID>/marketing.md` で管理する。
