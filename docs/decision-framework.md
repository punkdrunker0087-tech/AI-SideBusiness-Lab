# decision-framework.md — 数値ベースの意思決定基準

このドキュメントは、CEOが「勘」ではなく「ルール」で経営判断を行うための
アルゴリズムを定義する。CEOはGo/No-Go、投資、撤退の判断において、必ず
このフレームワークを基準にする。評価そのものを行うのは事業評価部
（投資委員会）だが、評価に使うロジックはこのドキュメントを正とする。

---

## 1. 新規案件の評価（8軸スコアリング）

8つの評価軸を0〜100点で採点し、重み付け合計で総合スコア（0〜100点）を
算出する。採点・判定は事業評価部が行い、CEOへ判定案として提出する。

| 評価軸 | 説明 | 重み |
|---|---|---|
| 市場性 | 市場規模（TAM/SAM/SOM）・需要の大きさ | 20% |
| 収益性 | 見込める利益率・単価・課金の成立しやすさ | 20% |
| 競争優位性 | 差別化のしやすさ、競合の少なさ・弱さ | 15% |
| 開発容易性 | 週末稼働・少人数で開発できるか | 10% |
| AI自動化率 | AIエージェントだけでどれだけ工程を回せるか | 10% |
| リスク | 法務・運用・レピュテーションリスクの小ささ（高スコア＝低リスク） | 10% |
| Evidence品質 | 市場調査部が集めたEvidenceの信頼度（後述の計算式） | 10% |
| 戦略適合性 | CLAUDE.mdの経営理念・経営原則・会社の方針との整合性 | 5% |

総合スコア = Σ（各軸の点数 × 重み）　※0〜100点

### Evidence品質スコアの計算式

市場調査部がEvidenceに付けたランク（`Research.md`のEvidence Ranking:
A=5, B=4, C=3, D=2, E=1）を用いて、以下で算出する。

```
Evidence品質スコア = (Σ 各Evidenceのランク重み) ÷ (Evidence件数 × 5) × 100
```

事業評価部は、市場調査部のConfidence Score（自己評価）をそのまま採用せず、
必ずこの計算式でEvidence品質を再計算する。

### 判定基準（5段階）

| 総合スコア | 評価 | 判定 | Portfolio Rule上のアクション |
|---|---|---|---|
| 85〜100 | ★★★★★ | Strong Go | 即着手 |
| 70〜84 | ★★★★☆ | Go | 着手候補（アクティブ枠が空き次第着手） |
| 55〜69 | ★★★☆☆ | Conditional Go | 保留（2週間以内に改善案を提出しなければPivot扱い） |
| 40〜54 | ★★☆☆☆ | Pivot | 改善（方向転換を検討し再調査） |
| 0〜39 | ★☆☆☆☆ | No-Go | 終了（30日保存後アーカイブ） |

CEOが即断で着手してよいのはStrong Goのみ。それ以外の運用ルール
（同時進行数の上限・改善案の提出期限・アーカイブまでの期間など）は
`docs/portfolio-strategy.md` を正とする。

### 強制No-Goゲート（総合スコアに関わらず適用）

- 法務部が「重大な法的リスクあり」と判定した場合
- 検証部（Validation）がREWORKと判定したまま是正されない場合
  （品質保証を通っていないEvidenceでは評価そのものが成立しない）
- 開発容易性が0〜10点（週末稼働では実装不可能）の場合

## 2. 運用中案件の継続／改善／撤退判定

分析部が毎サイクル（週次目安）で以下を計測し、PM経由でCEOに報告する。

| 状態 | 条件 | 判定 |
|---|---|---|
| 継続 | 利益率がプラス、またはKPIが右肩上がり | そのまま継続 |
| 改善 | 利益率が横ばい〜微減、または一部KPIが未達 | 改善部へ改善施策を依頼 |
| 撤退検討 | 3サイクル連続で利益率がマイナス、かつ改善施策を2回実施しても回復しない | CEOが撤退を判断 |
| 即撤退 | 重大な法的リスクが発覚、または損失が継続コストを大幅に超過 | CEOが即時撤退を判断 |

継続中案件の総合スコアも、docs/score-history.mdの記録に従って推移を追跡し、
改善施策が実際にスコアを押し上げているかを可視化する。

## 3. 判断の原則

- CEOは上記の数値基準に基づいて判断し、基準に定めのない「勘」による判断は行わない
- 評価軸・重み・閾値の変更はCEOのみが行え、変更した場合は理由と変更履歴を残す
- 既存の評価軸でカバーできない要素が出てきた場合、事業評価部が新しい軸を提案し、
  CEOが承認して初めてこのドキュメントに追加する
