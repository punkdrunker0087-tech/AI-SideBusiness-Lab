# 週次まとめ動画 SOP（EXP-008）

制定: 2026-07-24（CEO指示）／改定: 2026-07-24（「まとめ」から「研究レポート」へ再設計。週報パッケージ化）

`BRAND.md`のコンテンツ4役割のうち**Depth（深い理解）**を担う動画。
毎日のショートが「今日」を記録するのに対し、週次動画は「今週」を
振り返り、視聴者を研究の同伴者にする。

## 核となる設計方針: 「まとめ」ではなく「研究レポート」

**「今週やったこと」の羅列（日記）にはしない。** 既存視聴者向けの
振り返りに留まると、新規視聴者に価値が伝わらず検索にも乗りにくい。
毎週を**「仮説→実装→結果→期待と違った点→次の仮説」という1本の研究**
として構成する。

**成功談より「仮説→失敗→改善」を優先する。** 例（Week1想定）:

> 今週は背景アニメーションを追加しました。期待したのは視聴維持率の
> 改善です。ただし現時点ではAnalyticsが取得できないため、改善したとは
> 言えません。来週データを取得し検証します。

このように「まだ分からない」を隠さず語ることが、研究チャンネルとしての
信頼を作る（`BRAND.md`のEvidence First・誇張なしと同じ原則）。

## Day 7は「Week 1 Design Review」（2026-07-24追加・EXP-011）

**Day 7（および7の倍数の各週）は「最初の週報を出す日」ではなく、
「研究システムそのものが設計どおり動くかを確認する日」として扱う。**
まだ動画の再生数・エンゲージメントの成果を評価する週ではない。
評価対象は研究テーマ（EXPの中身）ではなく、**研究所そのもの**——
以下の5領域を確認する:

1. **動画**: 研究レポート形式（仮説→実装→結果→期待と違った点→
   次の仮説）で構成できているか
2. **運用**: `ROUTINE.md`通りに日次・週次が回ったか、手作業はどれだけ
   残ったか（システム品質評価の5項目参照）
3. **ドキュメント**: `EXPERIMENTS.md`・`OBSERVATIONS.md`・`WEEKLY.md`・
   `ROUTINE.md`・`LESSONS.md`・`PRINCIPLES.md`の間に矛盾・重複がないか
4. **Lesson管理**: `LESSONS.md`の`Candidate`が適切に`Accepted`/
   `Rejected`/`Superseded`へ確定しているか
5. **Routine**: トリガーが`ROUTINE.md`参照方式のまま安定して発火して
   いるか

### Day 7における「成功」の定義（2026-07-24 CEO明確化）

**Day 7は「成功」を期待する日ではない。** 期待するのは、
**システムが、設計した通りに自分自身を評価できることそのもの。**

例えば以下の結果になったとしても、それは十分に成功である
（すべて事前に定めたルールに従って一貫してレビューが行われたなら）:

- `LESSON-004`が`Rejected`だった
- `Principle`が0件しか昇格しなかった（`PRINCIPLES.md`の厳格な条件通り）
- Product改善は予定どおりDay 6で再開した
- `Portfolio Balance`が「正常」判定だった

**逆に避けるべきなのは、良い結果を出すためにルールを当日に読み替える
こと**（AI CEO憲法の観測結果はOSより上位・研究者バイアス回避の原則と
同じ）。

### Day 7で確認するたった1つのこと（2026-07-24追加）

評価項目は他にも多数あるが、最後に確認するのはこれだけでよい:

> **Day 7終了時点で、Day 8に何をすればいいか迷っていないか。**

迷っていなければ、研究OSは機能している。「次は何を改善しよう」という
会議が始まるなら、まだOSより人間の判断に依存している。良いOSは、
次の一歩まで自然に導く。

## 仕様

- **形式**: 通常動画（Shortsではない）・3〜5分
- **投稿タイミング**: Day 7の倍数（Day 7・14・21・28）に毎週
- **対象期間**: 直近7日分（例: Week1はDay1〜7）
- **サムネイル必須**: 通常動画はShortsと違い動画フレームの自動サムネに
  頼れない。`pipeline/weekly/`でHTML/CSSから1280×720のサムネイルを生成する
  （実写・写真は使わずブランド配色のテキストベース）

## 台本構成（固定タイムライン・毎週同じ構成でよい）

| 目安時刻 | パート | 内容 |
|---|---|---|
| 00:00 | 今週のテーマ | 今週、何を検証したか（1文で言い切る） |
| 00:20 | 実験内容 | 何を検証したか（仮説・EXP番号） |
| 01:20 | 実装した内容 | 何を変更したか（技術的な変更点） |
| 02:00 | 結果 | 何が起きたか（実測数値） |
| 03:00 | 分かったこと | **期待と違った点を含めて正直に。** 「改善したとは言えない」も可 |
| 04:00 | 来週試すこと | 次の仮説（新規EXP番号を予告） |

## 固定コーナー（毎回同じフォーマット）

動画の終盤に必ず入れる3ブロック:

1. **今週の数字**: 投稿本数・総再生数・登録者・CTR（取得できれば）・
   進行中の実験数
2. **今週の学び**: 一番大きかった気付きを1つに絞る（複数挙げない）
3. **来週の実験**: `EXP-00N「〇〇を検証します」`の形式で予告し、
   `EXPERIMENTS.md`にPlanned状態として新規登録する

## 週報パッケージ（5点セット・2026-07-24追加）

週次動画を単体で作らず、同じ取材内容から5つの成果物をまとめて生成する。
1回の実験を複数メディアへ展開することで資産性を高める。**2〜5は
ドラフトのみ生成し、投稿・公開はCEO（L3）が行う。**

| # | 成果物 | 内容 | 保存先 |
|---|---|---|---|
| 1 | YouTube動画（3〜5分） | 上記タイムラインでストーリー形式 | `week-0N/weekN.mp4` |
| 2 | Xスレッド | 動画の要点を10投稿前後に要約 | `week-0N/x-thread.md` |
| 3 | note記事 | 実験の背景・実装・結果・考察を詳述 | `week-0N/note-article.md` |
| 4 | 変更履歴 | `EXPERIMENTS.md`自体が変更履歴を兼ねる。専用ファイルは
作らず、対象週に追加・更新されたEXP番号一覧を`report.md`冒頭にまとめる | `week-0N/report.md` |
| 5 | 来週の実験計画 | 次の仮説と成功判定基準。`EXPERIMENTS.md`にPlannedとして
新規登録（実行はまだしない） | `EXPERIMENTS.md` |

## ソース

- `EXPERIMENTS.md`（対象週のEXP項目）
- `OBSERVATIONS.md`（Observation項目）
- `REJECTION-LOG.md`
- `days/day-0N/report.md`（対象週の7日分）

## 生成方法

台本はショートと同じ`pipeline/make-short-cloud.mjs`＋`voicevox-local/
template.html`を流用できる（シーン数を増やし尺を伸ばすだけで技術的な
変更は不要）。サムネイルのみ専用パイプライン:

```bash
cd media-division/pipeline/weekly
node make-thumbnail.mjs thumb.json thumbnail.png
```

`thumb.json`の形式:
```json
{
  "week": "WEEK 1",
  "headline": "見出し（<span class=\"accent\">...</span>で強調可）",
  "stats": [{"num": "5", "label": "本の実録"}, {"num": "0.7%", "label": "介入率まで低下"}]
}
```

## Research Review（毎週末・必須・2026-07-24追加）

週報パッケージの最後に必ず実行し、`week-0N/report.md`に記録する。
これがあることで、毎週の実験が独立せず、一つの研究プロジェクトとして
連続性を持つ。翌週のRoutine実行時（Day7の倍数の日）は、前週の
Research Reviewを参照してから当週の台本を作る。

```
Research Review — Week N

【最初に・固定】今週、この研究はユーザーにどんな価値を追加したか？
  （Portfolio Balance・Cost・Layer・Statusはすべて内部指標である。
  最終的に目指すのは外部成果——動画が見られる・登録者が増える・
  視聴維持率が上がる・収益につながる。答えが「ガバナンスを改善した」
  だけの週が続くなら黄色信号。「〇〇を追加し、視覚体験を改善した
  （効果は検証中）」のように、内部改善であっても外部成果への接続を
  明示する）

今週の仮説は正しかったか？: YES / NO / 保留
判定理由タグ: （下表から1つ以上）
その理由:
次週の仮説:
実験番号: EXP-0NN
引き継ぐ課題:
中止する課題:
研究プロセス自体の改善点: 今週の実験だけでなく、研究の進め方そのものに
  改善点はあったか？
ルール自身のレビュー: 今週新しく作ったルールで、実際に運用してみると
  不都合が見つかったものはあるか？（LESSONS.mdのCandidate状態の教訓を
  Accepted/Rejected/Supersededに確定する場もここ）
Portfolio Balance: 十分 / やや偏り / 大きく偏り
  （今週のEXPのLayer×Cost分布を見て判定。数字を作ることが目的では
  ない。偏りに気付ければ十分）
最後の問い: 来週、この改善をやらなかったら何が失われるか？
  （Governance改善を1週間延期しても問題ないなら後回しでよい。
  Product改善を止めるとチャンネルが止まるなら、翌週はProductへ
  重心を戻す。この問いが自然な優先順位の再調整をもたらす）

【最後に・固定】整合性チェック: 冒頭の「ユーザー価値」の回答と、
  このレビューの結論は矛盾していないか？
  （例: 冒頭で「今週はユーザー価値を増やせなかった」と書いたのに、
  結論が「今週は大成功だった」になっていたら、内部指標に引っ張られて
  いる。逆に、冒頭から結論まで評価が一貫していれば、たとえ成果が
  ゼロでもそのレビューは成功——`MISSION.md`参照）
```

### Portfolio Balanceの判定基準（参考・2026-07-24 CEO評価）

短期の偏りは正常な場合がある。例えばDay 6〜7直前（研究所そのものを
設計する期間）にGovernance層のコストが増えるのは目的に対して自然。
**「やや偏り」「大きく偏り」の判定は、単発の週ではなく複数週の
継続性で見る。** 目安: 同じ層への偏りが**3週間続いたら「大きく偏り」**
として扱い、翌週の重心調整を検討する。単発週の偏りだけで警報を
出さない（過剰反応を避ける）。

### 判定理由タグ（2026-07-24追加）

YES/NO/保留だけでなく、理由をタグ化して残す。数十件たまれば
「効果がなかった実験」だけでなく「そもそも測定できなかった実験」が
何件あるかを分析でき、研究のボトルネックが見える。

| タグ | 意味 |
|---|---|
| DATA | データ不足で判定保留 |
| EFFECT | 期待した効果を確認 |
| NO_EFFECT | 効果なし |
| REGRESSION | 悪化 |
| IMPLEMENTATION | 実装不備で再検証 |
| EXTERNAL | アルゴリズム変更など外部要因 |
| INCONCLUSIVE | 他要因と切り分け不能 |

例（EXP-007・Week1想定）: 判定＝保留（DATA） ——
視聴維持率のAnalyticsデータが取得できず、効果の有無を判定する材料が
揃っていないため

- 「保留」は許容する（Analyticsが取得できない等、判断材料が
  揃わない場合はYES/NOを断定しない。憲法第4条）
- 「研究プロセス自体の改善点」の問いにより、動画生成システムだけでなく
  研究の進め方そのものを継続的に改善するメタPDCAを回す
- 「中止する課題」は削除ではなく、`EXPERIMENTS.md`側でArchived/
  Supersededのステータスに変更して残す（憲法第2条・第3条）

## システム品質評価（毎週・Research Reviewと同時に実施・2026-07-24追加）

**動画の品質と運用システムの品質を分けて評価する。** Day 7は最初の
週報を出す日であると同時に、「週次システム自体が機能するか」を検証する
最初の実験日でもある。5項目×5点満点（計25点）を`week-0N/report.md`の
Research Reviewに併記する。

| 項目 | 確認内容 |
|---|---|
| 完全性（1-5） | 必要な成果物（動画・サムネイル・Xスレッド案・note記事案・
実験計画）はすべて生成されたか |
| 一貫性（1-5） | `ROUTINE.md`・`WEEKLY.md`・`EXPERIMENTS.md`の内容に矛盾は
ないか（記載の重複・古い情報の残存を含む） |
| 再現性（1-5） | 同じ入力（対象週の記録）なら同じ成果物を再生成できるか |
| 自動化率（1-5） | 手作業（L1〜L3の人間介入）はどれだけ残ったか。
少ないほど高得点 |
| 改善点（1-5） | 来週までに修正すべき点が明確に言語化できているか |

低いスコアが出た項目は、次の「研究プロセス自体の改善点」欄で
具体的な改善提案に落とし込む。
