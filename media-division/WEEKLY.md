# 週次まとめ動画 SOP（EXP-008）

制定: 2026-07-24（CEO指示）／改定: 2026-07-24（「まとめ」から「研究レポート」へ再設計。週報パッケージ化）

`BRAND.md`のコンテンツ4役割のうち**Depth（深い理解）**を担う動画。
毎日のショートが「今日」を記録するのに対し、週次動画は「今週」を
振り返り、視聴者を研究の同伴者にする。

## 核となる設計方針: 「まとめ」ではなく「研究レポート」

**「今週やったこと」の羅列（日記）にはしない。** 既存視聴者向けの
振り返りに留まると、新規視聴者に価値が伝わらず検索にも乗りにくい。
毎週を**「仮説→実装→結果→期待と違った点→次の仮説」という1本の研究**
として構成する。

**成功談より「仮説→失敗→改善」を優先する。** 例（Week1想定）:

> 今週は背景アニメーションを追加しました。期待したのは視聴維持率の
> 改善です。ただし現時点ではAnalyticsが取得できないため、改善したとは
> 言えません。来週データを取得し検証します。

このように「まだ分からない」を隠さず語ることが、研究チャンネルとしての
信頼を作る（`BRAND.md`のEvidence First・誇張なしと同じ原則）。

## 仕様

- **形式**: 通常動画（Shortsではない）・3〜5分
- **投稿タイミング**: Day 7の倍数（Day 7・14・21・28）に毎週
- **対象期間**: 直近7日分（例: Week1はDay1〜7）
- **サムネイル必須**: 通常動画はShortsと違い動画フレームの自動サムネに
  頼れない。`pipeline/weekly/`でHTML/CSSから1280×720のサムネイルを生成する
  （実写・写真は使わずブランド配色のテキストベース）

## 台本構成（固定タイムライン・毎週同じ構成でよい）

| 目安時刻 | パート | 内容 |
|---|---|---|
| 00:00 | 今週のテーマ | 今週、何を検証したか（1文で言い切る） |
| 00:20 | 実験内容 | 何を検証したか（仮説・EXP番号） |
| 01:20 | 実装した内容 | 何を変更したか（技術的な変更点） |
| 02:00 | 結果 | 何が起きたか（実測数値） |
| 03:00 | 分かったこと | **期待と違った点を含めて正直に。** 「改善したとは言えない」も可 |
| 04:00 | 来週試すこと | 次の仮説（新規EXP番号を予告） |

## 固定コーナー（毎回同じフォーマット）

動画の終盤に必ず入れる3ブロック:

1. **今週の数字**: 投稿本数・総再生数・登録者・CTR（取得できれば）・
   進行中の実験数
2. **今週の学び**: 一番大きかった気付きを1つに絞る（複数挙げない）
3. **来週の実験**: `EXP-00N「〇〇を検証します」`の形式で予告し、
   `EXPERIMENTS.md`にPlanned状態として新規登録する

## 週報パッケージ（5点セット・2026-07-24追加）

週次動画を単体で作らず、同じ取材内容から5つの成果物をまとめて生成する。
1回の実験を複数メディアへ展開することで資産性を高める。**2〜5は
ドラフトのみ生成し、投稿・公開はCEO（L3）が行う。**

| # | 成果物 | 内容 | 保存先 |
|---|---|---|---|
| 1 | YouTube動画（3〜5分） | 上記タイムラインでストーリー形式 | `week-0N/weekN.mp4` |
| 2 | Xスレッド | 動画の要点を10投稿前後に要約 | `week-0N/x-thread.md` |
| 3 | note記事 | 実験の背景・実装・結果・考察を詳述 | `week-0N/note-article.md` |
| 4 | 変更履歴 | `EXPERIMENTS.md`自体が変更履歴を兼ねる。専用ファイルは
作らず、対象週に追加・更新されたEXP番号一覧を`report.md`冒頭にまとめる | `week-0N/report.md` |
| 5 | 来週の実験計画 | 次の仮説と成功判定基準。`EXPERIMENTS.md`にPlannedとして
新規登録（実行はまだしない） | `EXPERIMENTS.md` |

## ソース

- `EXPERIMENTS.md`（対象週のEXP項目）
- `OBSERVATIONS.md`（Observation項目）
- `REJECTION-LOG.md`
- `days/day-0N/report.md`（対象週の7日分）

## 生成方法

台本はショートと同じ`pipeline/make-short-cloud.mjs`＋`voicevox-local/
template.html`を流用できる（シーン数を増やし尺を伸ばすだけで技術的な
変更は不要）。サムネイルのみ専用パイプライン:

```bash
cd media-division/pipeline/weekly
node make-thumbnail.mjs thumb.json thumbnail.png
```

`thumb.json`の形式:
```json
{
  "week": "WEEK 1",
  "headline": "見出し（<span class=\"accent\">...</span>で強調可）",
  "stats": [{"num": "5", "label": "本の実録"}, {"num": "0.7%", "label": "介入率まで低下"}]
}
```

## Brand Guardian追加チェック（週次のみ）

通常のチェックリストに加えて:

- [ ] 「仮説→実装→結果→期待と違った点→次の仮説」の構成になっているか
  （単なる出来事の羅列になっていないか）
- [ ] 失敗・未検証・伸び悩みを隠さず含めているか（成功談だけで
  終わっていないか）
- [ ] 複数日のデータを正確に集計しているか（誇張・切り取りがないか）
- [ ] サムネイルの数値が本編・台帳の実測値と一致しているか（Promise Gap防止）
- [ ] Xスレッド・note記事も本編と矛盾する数値・誇張がないか

## Research Review（毎週末・必須・2026-07-24追加）

週報パッケージの最後に必ず実行し、`week-0N/report.md`に記録する。
これがあることで、毎週の実験が独立せず、一つの研究プロジェクトとして
連続性を持つ。翌週のRoutine実行時（Day7の倍数の日）は、前週の
Research Reviewを参照してから当週の台本を作る。

```
Research Review — Week N

今週の仮説は正しかったか？: YES / NO / 保留
その理由:
次週の仮説:
実験番号: EXP-0NN
引き継ぐ課題:
中止する課題:
```

- 「保留」は許容する（Analyticsが取得できない等、判断材料が
  揃わない場合はYES/NOを断定しない。憲法第4条）
- 「中止する課題」は削除ではなく、`EXPERIMENTS.md`側でArchived/
  Supersededのステータスに変更して残す（憲法第2条・第3条）

## 年間資産としての位置づけ

52週続けば、Week1〜Week52がそのまま「AI動画自動生成プロジェクトの開発
ログ」になる。このログ自体が、書籍・note連載・講座・OSS的ドキュメント・
技術登壇に展開できる一次情報になる（`ORG.md`の資産化カスケードの
週次版）。
