## EXP-001 クラウド実行環境だけでショート動画を全自動生成できるか？

- 日付: 2026-07-19
- 仮説: HTML/CSS→ブラウザ録画→ffmpegの経路なら、動画編集ソフトなしで縦型ショートを生成できる
- 方法: 1080×1920のHTMLアニメーションをChromiumで録画し、ffmpegでH.264 mp4化。BGMはffmpegのシンセ合成
- 結果: **成功。** 12秒サンプルを人間の作業ゼロ・約2分で生成
- 発見した制約:
  - Playwright付属ffmpegはVP8/webm専用 → apt導入のフル版ffmpegが必須
- 資産: `pipeline/`（SOP・テンプレート・録画スクリプト）
- コンテンツ化候補: ショート「編集ソフトを1回も開かずに動画を作る会社」

## EXP-002 音声まで一括生成できるか？

- 日付: 2026-07-19
- 仮説: TTSをパイプラインに組み込めば、台本→音声→同期した映像まで一括生成できる
- 方法: edge-tts（ニューラル）とOpen JTalk（完全ローカル）を実測。音声の実測秒数にCSSタイムラインを同期
- 結果: **成功。** 「AI社長の朝礼」21秒サンプルを音声つきで一括生成
- 発見した制約（= 一次情報）:
  1. **VOICEVOXは取得不可。** 実行環境のGitHubアクセスがセッション登録リポジトリに制限されており、AI社長は自力でキャラ音声を調達できない。回避策: 人間のPCでの一回のセットアップ、またはクラウドTTSのAPIキーを人間が設定
  2. **edge-ttsは素のままではTLSエラー。** プロキシCAをcertifiに追加して回避（手順は`pipeline/README.md`）
- 資産: `pipeline/README.md`のTTS比較表・セットアップ手順
- コンテンツ化候補: ショート「AI社長、声を買いに行って門前払いされる」

## EXP-003 人間がPCにVOICEVOXを入れれば、クラウドのAI社員から使えるか？

- 日付: 2026-07-19
- 経緯: EXP-002でVOICEVOXの自力調達がブロックされたため、人間（代表）が
  自分のPCにVOICEVOXをインストールし、AI社員に接続確認を指示
- 方法: クラウド実行環境から 127.0.0.1:50021（VOICEVOXエンジンの待機ポート）へ
  接続テスト
- 結果: **不可。** AI社員が動くクラウドコンテナと人間のPCは別マシンであり、
  127.0.0.1は互いに自分自身を指す。「インストールしたのに繋がらない」は
  故障ではなく、環境の隔離という構造的制約
- 発見した制約（= 一次情報）:
  - AIと人間が「同じ会社」でも、動いている場所は物理的に別。声の工場を
    どちらのマシンに置くかは組織設計の問題になる
  - トンネル（ngrok等）でPCのエンジンを公開すれば接続自体は可能だが、
    ローカルサービスをインターネットに晒すためセキュリティ上推奨しない。
    PCの電源にも依存する
- 回避策（採用）: **分業。** 台本・テンプレート・タイミング設計はクラウドの
  AI社員が用意し、音声合成〜投稿は人間のPCで実行する。実行手順を
  `pipeline/voicevox-local/` にキット化（合成スクリプト+セットアップ手順）
- 資産: `pipeline/voicevox-local/`（voicevox.mjs・README）
- コンテンツ化候補: ショート「AI社長と人間、同じ会社なのに住む世界が違った」

## EXP-005 AI運営の会社は、30日間コンテンツを出し続けられるか？（旗艦実験）

- Status: **Running**（Day 1 = 2026-07-20、07:00 JST予約投稿をもって開始確定）
- 位置づけ: **当部門の本命実験。** 「AIが動画を作れるか」（EXP-001〜004）は
  前座であり、本丸は「AIという組織が継続できるか」の検証
- 主題（共同レビュアーによる再定義・2026-07-19）: 本実験の真の問いは
  「AIでショート動画を作れるか」ではなく、
  **「Evidence Firstという文化は、30日間の現実に耐えられるか」**である
- 日次報告の形式（動画の出来栄えより先にこの3点）:
  1. 今日の仮説
  2. 今日の観測
  3. 今日、OSはどこで現実にぶつかったか
- 問いの分解（毎日記録する観察項目）:
  1. 毎日どこで詰まったのか（ボトルネックの所在）
  2. 人間は何回・どのレベルで介入したのか（`ORG.md`の介入レベルL0〜L3で分類）
  3. AI CEOはどんな改善指示を出したのか
  4. SOPは何回更新されたのか
- 成功条件（2つとも満たして成功。片方では不成立）:
  1. **継続**: 30日間、1日1本以上の公開承認可能なコンテンツを提出し続けること
     （公開するかは人間の判断。提出が途切れた日は「なぜ途切れたか」を記録すれば
     実験自体は継続する — 途切れの記録も成果物である）
  2. **説明可能性**: 30日終了時に、L1〜L3介入の質と量を定量的に説明できること。
     具体的には「介入率は◯%でした。介入理由の上位3つはこれです。次の30日では
     この改善を試します」という形式のレポートが書けること。
     自動運転のディスエンゲージメント報告と同型であり、これが書けて初めて
     30日間は運営記録ではなく**誰かが再現できる研究成果**になる
- この実験の設計思想: **会社を運営すること自体がコンテンツ制作プロセスである。**
  観察項目1〜4がそのまま翌日のショート・週次のnoteの素材になる
- 完了時の資産化: 30日レポート（note/Zenn長文）、人間介入率の推移グラフ、
  更新されたSOP群、失敗事例集

## EXP-006 動画の文字は正しい日本語フォントでレンダリングされているか？

- 日付: 2026-07-23（CEOの指摘で発覚）
- 経緯: CEOから「動画の書体が中華漢字になっていないか、日本人向けの
  書体かチェックしてくれ」と指摘を受け、初めて検証した
- 方法: `fc-match`でこの環境のフォント解決を確認。`template.html`の
  CSSは`"Noto Sans CJK JP", "Noto Sans JP", sans-serif`を指定して
  いたが、いずれも未インストールで、最終的に**中華圏向けフォント
  「WenQuanYi Zen Hei」にフォールバック**していたことが判明
- 検証: フォント導入前後で同一シーンをレンダリングし、フレームを
  差分比較。ほぼ全ての文字で字形が変化しており、フォールバックが
  実際に発生していたことを画像で確認
- 結果: **失敗。** Day 1・Day 2・Day 3は中華圏フォントのまま
  **既に公開済み**。Day 4はレビュー段階（未投稿）だったため、
  `fonts-noto-cjk`導入後に再レンダリングして正しい日本語フォントに
  差し替え、公開前に修正できた。**Day 4は2026-07-23 16:00に
  フォント修正版として公開確認済み** — EXP-005史上初めて
  「公開前に発見した欠陥を直してから出せた」ケースになった
- 根本原因: パイプラインの環境セットアップ手順
  （`pipeline/README.md`）に、そもそも日本語フォントの導入手順が
  一度も含まれていなかった。ffmpeg・edge-tts・playwright-coreは
  毎回導入する運用だったが、フォントだけ抜け落ちていた
- 修正: `apt-get install -y fonts-noto-cjk`を必須セットアップ手順に追加
- **CEO判断（2026-07-23）: Day 1〜3はそのまま残す。** 再投稿による
  視聴回数・エンゲージメントの継続性喪失を避けることを優先。
  中華圏フォントで公開されていた事実は削除せず、この記録として残す
  （憲法第2条）。Day 4以降は正しい日本語フォントで統一される
- 資産: `pipeline/README.md`「フォントに関する注意」節
- コンテンツ化候補: 「AI社長、4日間気づかずに中国語フォントで喋っていた」

## EXP-009 Routineをバージョン管理された仕様書参照方式に変更すると、変更履歴の追跡性は改善するか？

- 日付: 2026-07-24（CEO指示）
- 経緯: EXP-008（週次動画の再設計）に伴いRoutineトリガーを
  delete_trigger→create_triggerで作り直そうとしたところ、CEOから
  「削除→再作成」ではなく「バージョン管理された運用」にする指示を受けた
- **変更理由**: トリガー本文に実行ロジックを直接書き込む方式だと、
  変更するたびに削除・再作成が必要で、「いつ・なぜ変えたか」の履歴が
  Routine側に残らない（トリガーの更新履歴とEXPERIMENTS.mdの記録が分離
  してしまう）
- **変更内容**: `ROUTINE.md`を新設し、Routineの実行内容（日次手順・
  週次手順・共通の前提）とバージョン履歴（v1: 日次ショートのみ→v2:
  週次研究レポート・週報パッケージ・Research Reviewを追加）を記録する
  唯一の正典とした。トリガー本文は「ROUTINE.mdの最新版に従う」ことだけを
  指示し、実行ロジックの詳細を持たない
- **期待する効果**: 今後の運用変更は「ROUTINE.mdを更新→必要ならEXP番号で
  変更理由を記録」だけで完結し、トリガーの再作成頻度が下がる。変更履歴が
  git（ROUTINE.mdの差分）とEXPERIMENTS.md（変更理由・効果）の二重で
  追跡可能になる
- **ロールバック方法**: `ROUTINE.md`を削除し、トリガー本文に実行ロジックを
  再び直接埋め込む（delete_trigger→create_triggerで復元）。過去バージョン
  のロジックはEXP-005初期のRoutine作成コミット履歴（git log）から復元可能
- 資産: `ROUTINE.md`（Routine仕様書・バージョン履歴つき）
- **同日追記（2026-07-24）**: Research Reviewの「YES/NO/保留」に
  判定理由タグ（DATA/EFFECT/NO_EFFECT/REGRESSION/IMPLEMENTATION/
  EXTERNAL/INCONCLUSIVE）を追加し、CEOの指摘で気づいた重複も解消した
  ——`ROUTINE.md`と`WEEKLY.md`の両方にResearch Review書式を書いており、
  Single Source of Truthの原則に反していた（`WEEKLY.md`に一本化）。
  さらに「今週の実験だけでなく、研究プロセス自体にも改善点はあったか？」
  を固定質問として追加し、動画生成システムだけでなく研究の進め方
  そのものを継続改善するメタPDCAを組み込んだ

## 記録ルール

- 各実験は 問い / 仮説 / 方法 / 結果 / 発見した制約 / 資産 / コンテンツ化候補 を残す
- 「発見した制約」は失敗ではなく収集物として扱う（研究KPI: AIの失敗事例収集数）
- アーカイブ部は本台帳の番号を主キーに全成果物をタグ付けする
- **成功を祝わない。** Success/Failed/Completedは状態であって評価ではない。
  研究記録の言葉は中立にする — Successは「仮説は現時点では支持された」、
  Failedは「仮説は棄却された」と書く。感情は自由だが、台帳には持ち込まない
- **反証ログを残す**（`REJECTION-LOG.md`）。AI CEOの提案が採用されなかった
  場合、提案内容と不採用の理由を記録する。採用しなかった仮説は、
  後日「今なら成立する」と再評価できる資産である
- **観測者効果を忘れない。** 台帳に書かれることを意識して、AI・人間双方の
  行動が変わる（判断が慎重になる、介入率を下げたい意識が判断に影響する、
  「コンテンツになる失敗」を無意識に選ぶ等）。これはノイズではなく
  観測対象であり、気づいたら `観測者効果` タグを付けて記録する
