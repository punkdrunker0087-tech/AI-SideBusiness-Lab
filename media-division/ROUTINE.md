# EXP-005 Routine 仕様書

**現在のバージョン: v2**（2026-07-24〜）

毎朝05:00 JSTに発火するRoutine（Claude Code Remoteのtrigger）が実際に
何をするかを定める正典。**トリガー本文には実行ロジックを直接埋め込まず、
本仕様書の最新版に従うことだけを指示する。** これにより、運用手順を
変更するたびにトリガーを作り直す必要が減り、「いつ・なぜ変えたか」を
本ファイルのバージョン履歴とEXPERIMENTS.mdの両方で追跡できる。

## 共通の前提

- **投稿・公開は絶対にしない（L3は人間の専権）**。提出物をレビュー待ちに
  置くまでが仕事
- Day 1 = 2026-07-20（07:00 JST予約投稿）。Day N の日付 = Day1の日付 +
  (N-1)日
- AI開示設定は運用ルールとして確定済み（2026-07-24）。日次の確認事項からは
  除外してよい
- 未消化（未投稿・未報告）の提出物が2日分以上溜まっている場合は新規生成
  せず、その旨だけ伝える

## 日次手順（毎日）

1. `git pull origin claude/limit-lab-company-setup-bud3og`し、
   `EXPERIMENTS.md`と直近の`days/day-NN/`を確認する
2. 今日の日付からDay番号を計算する。Day1当日でまだ07:00の予約投稿が
   確認できない場合は生成せず終了する
3. Day Nが確認済みなら、当日分の実録ショートを生成する:
   - 環境セットアップは`pipeline/README.md`のSOP通り毎回行う
     （**`fonts-noto-cjk`の導入を絶対に省略しない**。EXP-006参照）
   - `pipeline/make-short-cloud.mjs`で生成後、フレームを抽出して
     日本語フォントで正しくレンダリングされているか目視確認する
   - テーマは前日に実際に起きたことから選ぶ。同一構成の連投禁止
   - 背景モーション（EXP-007・`template.html`）はそのまま使う
   - `days/day-NN/publish.md`・`report.md`を作成し、タイトル・説明文・
     タグは**チャット本文にも全文を貼る**（ファイルアクセスを要求しない）
4. コミット・プッシュ・動画をSendUserFileで送付・簡潔に報告する

## 週次手順（Day番号が7の倍数の日、日次手順に加えて実行）

詳細仕様は`WEEKLY.md`。要約:

1. 直近7日分の`days/day-0N/report.md`・`EXPERIMENTS.md`・
   `OBSERVATIONS.md`・`REJECTION-LOG.md`・**`LESSONS.md`**（既存の教訓に
   照らして判断がぶれていないか確認する）から、「仮説→実装→結果→
   期待と違った点→次の仮説」の研究レポート台本を作成する（単なる
   出来事の羅列にしない）
2. 週報パッケージ5点セットを生成する（動画・Xスレッド案・note記事案・
   変更履歴・来週の実験計画）。動画とサムネイル以外はドラフトのみ
3. **Research Reviewとシステム品質評価を実行し、`week-0N/report.md`に
   記録する**（書式・判定理由タグ・評価5項目の定義は`WEEKLY.md`参照
   ——本仕様書では重複させない）。この記録は翌週のRoutine実行時に
   参照し、実験の連続性を保つ
4. Research Reviewの「研究プロセス自体の改善点」で挙がった提案のうち、
   一般性のあるものは`LESSONS.md`への昇格候補としてCEOに提示する
   （昇格の可否はCEOが判断。AIが独断で追記しない）
5. `media-division/weeks/week-0N/`へ保存し、コミット・プッシュ・
   動画とサムネイルを送付・報告する

## 本仕様書の改定について

改定は`EXPERIMENTS.md`にEXP番号を採番して記録する（運用方法そのものの
変更を実験として扱う。変更理由・期待する効果・ロールバック方法を含む）。
直近の改定はEXP-009参照。
