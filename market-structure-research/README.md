# Market Structure Research

> 研究・学習用。発注機能は持たない。

> ⚠️ **呼び方の訂正（査読コメント）**: このプロジェクトは「戦略開発
> フレームワーク」でも「アルファ探索基盤」でもない。実態に最も近い
> 呼び方は**Research Operating System**——仮説を作り、弱く表現し、
> 反証し、棄却を資産化し、再現可能に残すためのOS。個々のRQの結果
> より、この一連の仕組み（DSR/PBO・Evidence Level・Robustness・
> Research Unit・Negative Results）自体の方が価値が大きい。

## 目的（`RESEARCH_CHARTER.md`より）

**本プロジェクトの目的は「アルファを発見すること」ではなく、
「アルファ候補を誤って採択しないための研究基盤を構築すること」である。**
候補が却下され続けることは失敗ではない。「なぜ却下したかを第三者が
追跡・検証できない」ことが失敗。詳細・非目標・メタ整備の終了条件は
`RESEARCH_CHARTER.md`参照。

研究基盤が「使える状態」になった段階で、**基盤を「アルファ候補の
大量探索」に使うか、「市場構造の仮説検証」に使うか**という分岐点に
達し、後者を選んだ（ユーザー提案）。前者はAQM-01・過去の候補探索
（Momentum→IVOL→FX→Rate、EPS→Gross Profitability→需給→イベント）と
同じ「探索空間を広げればいつか当たる」構造に陥る危険がある。後者では
イベントは「アルファ候補」ではなく**観測対象**になる。この分岐に伴い、
名称も`alpha-research/`から`market-structure-research/`へ改名した
（目的が「アルファ探索」ではなく「市場構造の仮説検証」であることを
名前に一致させるため）。

## 3つのテーマに限定する（候補を無制限に集めない）

分岐後、候補を無制限に集めるのではなく、以下の3テーマに限定する
（詳細: `RESEARCH_CHARTER.md`）:

| テーマ | 問い | 現状 |
|---|---|---|
| Theme 1 | 価格はどの速度で情報を織り込むのか | RQ-001/PV-001で着手済み（下記） |
| Theme 2 | どんな主体が裁定を遅らせるのか | `ARBITRAGE_AGENT_MAP.md`の時間軸を今後実証 |
| Theme 3 | どんな制約が価格の歪みを残すのか | `ARBITRAGE_CONSTRAINT_MAP.md`の代理変数を今後実証 |

イベント（指数組入れ・政策保有株解消・決算等）は、この3テーマを
検証するための**実験材料**であり、それ自体を「良さそうだから採用」
する対象ではない。

## 中核ドキュメント（実装より先に整備する設計図）

これまでの検証（EPS修正率の実装・却下）を通じて、「思いついた候補を
実装する」のではなく、**先に研究の設計図を整備してから候補を選ぶ**
という進め方に転換した。中核となるドキュメントは以下の6つ:

| ドキュメント | 内容 |
|---|---|
| `RESEARCH_CHARTER.md` | プロジェクトの目的・非目標・メタ整備の終了条件・採択基準を固定する憲章 |
| `ALPHA_GENERATION_MAP.md` | 価格形成メカニズムを①新情報②需給③制度④行動⑤流動性に分類し、さらに情報・制約・摩擦という3つの根源的な力に抽象化した地図 |
| `ARBITRAGE_AGENT_MAP.md` | 市場参加者（個人・アクティブ投信・パッシブETF・HFT・年金等）ごとに、情報取得・判断能力・売買制約・裁定能力を整理し、「誰が・いつ知っていて動けないか」を時間軸込みで可視化 |
| `ARBITRAGE_CONSTRAINT_MAP.md` | なぜ歪みが解消されず残るのか、裁定コスト（資本制約・規制制約・執行コスト・組織制約）の構造を整理。情報/制約/摩擦をさらに細分化（測定可能な代理変数が伴う場合のみ使用）。「イベントスタディ病」を避けるため、イベントを裁定可能/裁定困難に事前分類する方針も明記 |
| `EVENT_EVALUATION_PIPELINE.md` | イベント研究をファクター研究と同等の厳格さ（事前仮説登録・イベント定義固定・Permutation Test・DSR・PBO・重複確認）で評価するパイプライン |
| `CANDIDATES.md` | 個別候補の検討記録。各候補について情報生成メカニズム・裁定制約・大型株での頑健性・現代文献での再現性・PIT構築可能性をスコアリング |
| `NEGATIVE_RESULTS.md` | 却下した候補・仮説の記録（検証設計・却下根拠の構造化タグ・Evidence Level）。このプロジェクトの最大の資産（下記参照） |
| `research_questions/RESEARCH_UNIT_TEMPLATE.md` | 個別の仮説検証を「テーマ→候補」ではなく「Research Question→検証→結論」という研究単位で記録するための固定テンプレート |
| `STRATEGY_CANDIDATE_QUEUE.md` | Decisionが「昇格」（E3-R2以上）になったRQだけを登録する最小限のキュー。フルパイプライン統合はまだ行わず、候補が5件前後溜まるかボトルネックが確認できるまではこの手動キューで十分とする |

候補を新たに検討する際は、この順序を守る:
**アルファ発生マップで分類 → 裁定主体マップで「誰が・いつ動けないか」を
特定 → 裁定制約マップで「コストの構造」を言語化（代理変数必須）→
イベント評価パイプラインで検証設計を固定 → 初めて候補として
`CANDIDATES.md`に追加 → 3段階ゲート（新規性→DSR/PBO→限界貢献度）で
検証**。この順序・各文書の役割分担を守ることが`RESEARCH_CHARTER.md`
の非目標（マップ整備自体の目的化）を避ける歯止めになる。

## 再現可能性の原則

候補を却下する際は、第三者が追跡・再現できる形で記録する
（`ARBITRAGE_CONSTRAINT_MAP.md`「再現可能性のための設計原則」参照）。
「効かなかった」という結果は、原因を断定せず、確認できた/できな
かったことの記述に留める（例: 「裁定主体が存在しない」ではなく
「今回の検証設計では裁定制約による持続的な超過収益を確認できなかった」）。

## サブテーマ: Point-in-Time財務データ（J-Quants）

現時点で唯一データ接続・実装まで進んだサブテーマ。

| ファイル | 内容 |
|---|---|
| `jquants_client.py` | J-Quants API(V2)クライアント。`x-api-key`認証、`/fins/summary`（決算短信サマリー）・`/equities/bars/daily`（株価四本値）取得 |
| `eps_revision_factor.py` | EPS修正率のPoint-in-Timeパネル構築（`DiscDate`基準、年度切り替わりは除外） |
| `eps_revision_gate_test.py` | EPS修正率の3段階ゲート検証（①相関→②DSR/PBO→③限界貢献度） |

### セットアップ

J-Quants APIはV2（2025-12-22以降に登録したアカウントはV2のみ利用可）。
認証はAPIキー方式（`x-api-key`ヘッダー）のみで、Googleアカウントでの
登録でも問題なく使える。ベースURLは実地の接続テストで
`https://api.jquants.com/v2`と確認済み（`api.jquants-pro.com`ではない
点に注意）。

```bash
cd market-structure-research
pip install -r requirements.txt

# マイページ「API Keys」から発行したキーを設定
export JQUANTS_API_KEY="発行したAPIキー"

# 接続確認
python jquants_client.py
```

Freeプランのデータ範囲は実地確認で**約2年分**（確認時点で
2024-04-30〜2026-04-30のローリングウィンドウ）。

### EPS修正率の検証結果（却下）

決算短信の`NxFEPS`（通期開示時の翌期予想）と`FEPS`（四半期開示時の
当期予想更新）を開示日順に連結し、同一年度内の修正率を計算。
`DiscDate`の翌日から値が有効になるPoint-in-Timeパネルを構築し、
3段階ゲートにかけた:

- **①新規性ゲート**: 既存5ファクター（Momentum・LowVol・Liquidity・
  FX・Rate）との相関は最大0.478（Liquidityと）。かろうじて合格
  したが際どい水準。
- **②単独DSR/PBOゲート**: 最良Sharpe+0.716、**DSR=0.736だが
  PBO=0.750**。DSR単独では合格に見えるが、PBOが極めて高く過学習の
  強い懸念があるため、DSR>0.5かつPBO<0.3の両方を要求する基準で
  **不合格**と判定した。
- **③限界貢献度ゲート**: ②不合格のため参考実施のみ。既存合成へ加える
  とむしろ悪化（Sharpe+1.11→+0.99）し、②と整合。

**却下の正確な理由**: 「大型株では裁定できない主体が存在しない」という
断定は誤り（`ARBITRAGE_AGENT_MAP.md`参照、アクティブ投信・海外クオンツ
等、情報取得○・判断能力○・裁定能力中〜高の主体は大型株にも存在する）。
正確には「今回の検証設計（大型株・高カバレッジユニバース・2年間）
では、裁定制約による持続的な超過収益を確認できなかった」。

## 反証可能仮説の検証: RQ-001（決算開示後の価格反応タイミング）とPV-001（ネガティブコントロール）

`research_questions/theme1_price_reaction_speed/RQ-001_earnings_
reaction_speed.md`で事前登録し、`earnings_reaction_speed_test.py`で
実行。日経225ユニバース・449件の決算開示イベントで検証した結果:

- **H2a（決算後20営業日の異常リターンの80%以上が最初の3営業日で
  実現する）: 棄却**（中央値0.585 < 0.80）
- **H2b（前倒し反応がランダム配置より有意に強い）: 支持**
  （permutation p<0.001）

「二つの異なる仮説が、それぞれ異なる判定になった」というのが正確な
記述であり、「中間的な結果」ではない。⚠️ なお、当初「決算情報の反映
速度」（情報処理速度）の研究として立てたが、Alternative Explanation
（流動性・銘柄サイズ・ボラティリティ等）を統制していないため、
「情報処理速度」を測ったとは言えないとRQ自体を「価格反応タイミング」
の研究に再定義した（Decision: 保留、詳細は同ファイル参照）。

続けて`research_questions/pipeline_validation/PV-001_negative_
control.md`で事前登録し、`negative_control_test.py`（RQ-001と同一の
パイプラインコードを再利用）でランダムな偽イベント日に対して検証
したところ、**有意な誤検出は確認されなかった**（p=0.253で非有意）。
これにより、RQ-001で検出した前倒し反応がパイプラインのアーティファ
クトではないことを確認した。

**この2本（RQ-001実イベントで検出・棄却判定ができる、PV-001偽イベント
で誤検出しない）が揃ったことで、研究基盤は「作る」段階から「使う」
段階に移行したと判断する。** Evidence Level: 両方ともE2-R0
（単一データセット・単一条件、感度分析は未実施）。

## RQ-002（流動性によるBlocking Strategy）

`research_questions/theme1_price_reaction_speed/RQ-002_liquidity_
blocking.md`。RQ-001のAlternative Explanationのうち最有力だった
「流動性」を、高流動性群・低流動性群に中央値分割して統制した。両群の
ratio点推定値はほぼ一致（中央値0.435 vs 0.442）したが、有意性は
低流動性群のみ（p=0.017、高流動性群はp=0.106）で、事前登録した
「想定していなかったパターン」に該当した。**Decision: 再実験**——
2群のp値を別々に見る設計では「効果が流動性に依存するか」を直接
検定できないという設計上の限界が判明したため、棄却でも支持でもなく、
2群の差を直接検定する設計（回帰・ブートストラップ）でのRQ-003を
次に予定する。

## RQ-003（流動性を連続変数として検証）

`research_questions/theme1_price_reaction_speed/RQ-003_liquidity_
continuous.md`。RQ-002で却下された「中央値2値分割」の代わりに、
流動性を連続変数のままSpearman順位相関で検証（RQ-001の449件全件を
使用、分割なし）。結果はSpearman ρ=-0.083、permutation両側p=0.060。
事前登録基準（p<0.05）ではH3棄却だが、基準をわずかに超えるのみの
境界線上の結果であり、**Decision: 保留**（確信を持った非有意では
ない。検出力不足・非常に小さい単調効果・単調関係の不在、いずれとも
整合的で本データでは区別できない）。⚠️ RQ-002（閾値分割）と
RQ-003（順位相関）は見た目は別の検証だが、**どちらも単調な関係
しか検定していない**という共通の弱点を持ち、完全に独立した2本の
証拠ではない（非単調な関係——例えば最下位分位でのみ効果がある形——
なら両方とも検出に失敗する）。また、下がったのは「流動性」の
説明力一般ではなく、**「流動性×本RQ-001のReaction Ratio」という
特定の組み合わせ**の説明力である。他のObservable（CAR(0,1)・ドリフト
速度・出来高ショック等）への流動性の説明力は未検討のまま。この特定の
組み合わせは次のRQの直接対象からは外れるが、次のRQ（RQ-004）は
「なぜそれが残る全Alternative Explanationの中で最も期待情報利得が
高いか」を冒頭に明記した上で選定する。

## PV-002（検出力の切り分け、RQ-004の前に実施）

`research_questions/pipeline_validation/PV-002_power_check.md`。
「Promoteが一度も出ないのは、Alternative Explanationが悪いのか、
それとも検証手法・サンプルサイズ自体が中程度の効果すら検出できない
のか」を切り分けるため、RQ-004（次の個別候補の検証）より先に実施。
n=449でのシミュレーションの結果、**RQ-003の設計空間（このObservable・
この統計量・このイベント数）に限って言えば、中程度の効果（ρ≥0.15）
に対する検出力は90%**で十分だが、RQ-003で実際に観測された小さな
相関（ρ=-0.083）は検出力が不安定な領域（ρ=0.10で検出力58%）に
あった（⚠️ 他のObservableに一般化される主張ではない）。
**Decision: 採用**——パイプラインがこの組み合わせにおいて構造的に
機能していないという懸念は後退し、次のAlternative Explanationを
同じObservable・統計量で検証することは正当化される。これにより
「流動性についてratio以外のObservableで焼き直す」という選択肢の
優先度は下がり、次のAlternative Explanationへ進む方が合理的になった。
同時に、RQ-002の中央値2値分割は本検証より検出力が低いことも確認され
（ただし2値化全般ではなく中央値分割という特定の操作化の話）、今後の
RQでは連続変数アプローチ（RQ-003方式）を優先する設計原則が確立
された。

## RQ-004（ボラティリティ、Observable自体の妥当性検証）

`research_questions/theme1_price_reaction_speed/RQ-004_volatility_
and_observable_validity.md`。銘柄サイズより先にボラティリティを
選定した理由: 銘柄サイズの検証は「Alternative Explanationが1つ
減る/残る」に留まるが、ボラティリティの検証は「Reaction Ratio
というObservable自体がボラティリティ駆動の測定ノイズに過ぎない
のでは」という、RQ-001〜003全体の土台に関わる懸念に答えられる
（査読指摘: コスト最小と情報利得最大は別）。結果はSpearman
ρ=-0.0175、p=0.717——RQ-003（流動性、p=0.060）より明確に非有意。
**Decision: 棄却**（今回定義したボラティリティProxy——イベント前
60営業日の実現ボラ——との単調関係という懸念を棄却。他の定義
（イベント直前ボラ・IV等）まで安全とは言えない）。事前登録した
「宿題」への回答: ボラティリティは強く相関しなかったため、RQ-001の
中核的発見は当初想定通り無傷、RQ-002・RQ-003への懸念も後退——想定
した非対称な
書き換えは「どちらもほぼ無傷」というより安心できる方向に確定した。
次はAlternative Explanationとして銘柄サイズの検証に進む。

## RQ-005（銘柄サイズ、安価なProxy群の絞り込み）

`research_questions/theme1_price_reaction_speed/RQ-005_size.md`。
時価総額（`ShOutFY`×イベント前終値）とratioのSpearman相関を検証。
結果はρ=-0.0168、p=0.716——RQ-004（ボラティリティ）とほぼ同一の、
明確な非有意。**Decision: 棄却**。「情報処理の遅延」という
Mechanism Candidateの最も標準的な支持経路（大型株ほど情報環境が
豊かで処理が速い）が1つ外れたことになる。これで流動性（境界線上）・
ボラティリティ（明確に非有意）・サイズ（明確に非有意）というTheme 1の
主要Alternative Explanationのうち3つが検証済みとなり、既存データで
安価に検証できる道筋はほぼ尽きた。残るアナリストカバレッジ・開示の
複雑さはいずれも着手コストが高く、Theme 1をこのまま安価な代理変数
だけで掘り下げる費用対効果は下がった——次に着手する前に、新規データ
取得へ投資するか、Theme 2・3へ研究の重心を移すかの判断が必要。

## RQ-006（Theme 1のシンセシス、新しい統計解析ではない）

`research_questions/theme1_price_reaction_speed/RQ-006_theme1_
synthesis.md`。新規データ取得にもTheme 2への移行にも進む前に、
Theme 1を整理する。**訂正**: これまで「Mechanism Candidateの
優先順位が下がった」と書いてきたが、正確には**「縮小したのは
Mechanism Candidate（情報処理の遅延）ではなく、安価に測定できる
Proxy群（サイズ・流動性・ボラティリティ）」**である。「情報処理の
遅延は支持されなかった」とはまだ言えない。Theme 1全体の
**Decision: 絞り込み（Narrow）**——棄却でも昇格でも保留でもない。
Theme 1でこれ以上やらないこと3点（サイズの焼き直し・流動性の中央値
分割への回帰・ボラティリティProxyの際限ない追加）を明文化。
**Closedではなく Archived**（現データセット・現Research Operating
Systemの範囲で前進の道が見えない状態。Point-in-Timeアナリスト
カバレッジ・板情報・機関投資家保有データ等が手に入れば再開しうる）。
**Theme 2への接続**: Theme 1は「情報環境」を否定していない、
Archivedになったのは「安価なProxy」である。したがってTheme 2は
「情報環境がダメだったから主体を見る」という消去法ではなく、独立に
「Reaction Timingに最も情報を与える制約は何か」という、Mechanism
Candidateを1つに固定しない開かれた問いから始める。

## Theme 2（Go with constraints、査読方針）

Theme 2着手は「Go」ではなく「Go with constraints」。Theme 1の最大の
敵がProxy≠Mechanismだったのに対し、Theme 2の最大の敵は
**Constraint≠Cause**——ある制約とReaction Timingが相関しても、
「その制約が価格反応を遅らせた」とは限らず、業種・指数採用・時価
総額等の共通の交絡要因を共有している可能性が常にある。新しい査読
項目は追加せず（テンプレートは凍結、既存のAlternative Explanation・
DAG・Blocking Strategyで十分扱える）、代わりに毎回この一問を繰り返す:
**「このConstraintが効いているという解釈以外に、同じ結果を説明でき
る競合説明は何か」**。Theme 2への期待値: 「新しいアルファ」ではなく
「市場参加者を情報処理主体ではなく制約を受ける主体として見ることで
研究全体の見方が変わるか」。最初のRQでは「制約が効くか」を証明する
のではなく、「どの制約のOperationalizationが情報を持ち、どれが
持たないかを切り分ける」という期待値で始める。

## RQ-007（Theme 2第1号: 決算開示種別という制約タイミングの操作化）

`research_questions/theme2_arbitrage_agents/RQ-007_disclosure_type_
constraint.md`。新規データ取得ゼロで検証できる制約操作化として、
決算開示種別（FY決算=年次ベンチマーク再構成・window dressing等の
制約タイミングと重なりやすい／四半期決算）を検証。RQ-002の教訓
（群間差そのものを検定する設計にする）を踏まえ、群ラベルの
シャッフルによる直接比較を実施。結果はFY平均0.817・Q平均0.744、
観測差0.073、p=0.342——明確な非有意。**Decision: 棄却**（この特定の
操作化を棄却。「機関投資家の制約が反応タイミングに影響する」という
Mechanism Candidateそのものは棄却しない、Theme 1の規律をそのまま
適用）。参考として確認したAlternative Explanation（情報量の違い、
`|CAR_20|`のFY/四半期差）も0.003とごく小さく、今回はConstraint≠Cause
の判断自体が不要だった。次に試す制約操作化（ベンチマーク乖離・
パッシブ資金・ETFフロー等）は、いずれも新規データまたはより手の
込んだProxy構築を要する。

## 知見（Findings）と投資候補（Investment Candidates）の違い

`RESEARCH_CHARTER.md`の通り、**E2-R0は「知見」であり「戦略候補」では
ない**。単一データセット・単一条件での支持/棄却は市場構造についての
知見の記録に過ぎず、実装や投資判断の根拠にはならない。「投資候補」に
昇格するには最低`E3-R2`（複数期間または複数ユニバースでの再現＋
感度分析済み）が必要。知見（E2-R0）が複数並んでも、「強そうなものを
選んで実装する」判断はDSR以前のmultiple testing問題であり禁止する。

## 最大の資産はネガティブ・リザルト・データベース

このプロジェクトを通じて分かったのは、イベント評価パイプラインという
「作ったもの」よりも、**「否定できる研究文化」を作ったこと**の方が
価値が大きいということ。EPS修正率・アクルーアル・粗利益収益性・
短期リバーサル・IVOL・スキューネス・共変動——「良かったもの」より
「捨てたもの」の方が多い。却下した候補とその理由は`NEGATIVE_RESULTS.md`
に蓄積し、将来同じ誤りを繰り返さないための資産として育てる。
