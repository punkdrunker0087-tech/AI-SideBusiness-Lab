# 研究・検証向けシステムアーキテクチャ（ペーパートレード基盤）

Data Providers → Validation → Feature Pipeline → Research Models →
Paper Trading Layer → Monitoring Dashboard の一気通貫パイプラインを実装。
**自動発注・ブローカー接続は一切含まない。**

> このリポジトリの`backtest-framework/BACKLOG.md`に積んであった
> 「Stage 7: ペーパートレード・ハーネス」の実装。研究・評価・監視までを
> 対象とし、実運用の自動売買システムとは明確に切り離す
> （`../macro-research/`と同じ境界線）。

## アーキテクチャ

```
Data Providers → Validation → Feature Pipeline → Research Models
  → Candidate Signals → Paper Trading Layer → Monitoring Dashboard
```

| コンポーネント | 責務 | 実装 |
|---|---|---|
| データ取得 | 収集・タイムスタンプ整合性・欠損検知・品質監視 | `data_provider.py` |
| Feature Pipeline | 特徴量生成・バージョン管理・再現性確保 | `feature_pipeline.py` |
| モデル評価 | 推論・信頼度出力・ドリフト監視 | `model_layer.py` |
| Paper Trading | 仮想約定・仮想損益・仮想コスト | `paper_trading.py` |
| Monitoring | データ遅延・モデルドリフト・仮想損益・推論時間・稼働状況 | `monitoring.py` |
| イベントログ | 全処理を時系列で記録し監査可能にする | `event_log.py` |
| アラート | データ更新停止・異常値・性能低下・障害の検知＋通知 | `alerts.py` |
| リコンシリエーション | 入力データ・モデル出力・仮想ポジションの整合性確認 | `reconciliation.py` |
| 実行制御 | 全体を1日単位のバッチとして駆動 | `orchestrator.py` |

## セットアップ & 実行

```bash
cd research-paper-trading-system
pip install -r requirements.txt
python demo.py
```

## 実装中に見つけたバグ（2件・教訓として記録）

### 1. PSI計算でゼロ幅ビンによる異常値膨張

決定木系モデルの確信度は離散値に集中しやすく、分位点でビン境界を作ると
境界が重複することがある。`np.histogram`はこれを無音でゼロ幅ビンとして
扱い、PSIが**7.5〜9.7**という明らかに異常な値になった。`np.unique`で
境界の重複を除去して修正。

### 2. サンプル数に対してビン数が多すぎる問題

1つ目を直した後も、ベースライン40件・直近20件を10ビンに割ると1ビン
あたり平均2〜4件しかなく、**偶然の空ビン**でPSIが依然として異常
（4〜9台）になった。サンプル数に応じてビン数を自動的に減らす
（1ビンあたり最低10件を確保）よう修正し、初めて解釈可能な値になった。

### 3. アラートの重複抑制がなく、同一条件で77回連続発報

ドリフト検出・ドローダウン超過が、条件が続く限り**毎日同じアラートを
繰り返す**設計になっており、120日のシミュレーションで77件のアラートが
発報された（実運用なら同じ理由で77回ポケベルが鳴ることになる）。
値が前回通知時から一定以上悪化した時だけ再通知する重複抑制
（`alerts._should_notify`）を追加し、**意味のある8件**まで削減した。

> 3件とも「動いた（エラーが出ない）」ことと「正しく・使い物になる」
> ことは別、という教訓。特に3件目は、コードは技術的に正しくても
> 運用設計として欠陥がある（アラート疲れ）という、この種のシステムに
> 固有の失敗モードだった。

## 実測結果（3銘柄・120日シミュレーション）

```
=== Monitoring Dashboard ===
データ遅延: 最新0日 (監視銘柄3件)
モデルドリフト(推論信頼度PSI): 4.224  ⚠️大きく変化
仮想損益: NAV 999,375→877,659  累積リターン-12.18%  最大DD-13.78%  約定360回  総コスト11,634
システム稼働状況: ○ 正常  (イベント総数858件)

リコンシリエーション:
  ポジション照合: ○ 内部状態とログが一致
  モデル判断照合: ○ モデル判断と約定が整合
```

- **モデルドリフトPSI=4.2**という大きな値は、バグではなく**本物の発見**:
  学習済みモデルを再学習せず120日間のOOS期間に適用し続けたところ、
  推論の確信度分布が期間の前半と後半で大きく変化した。これは静的モデルを
  監視なしで長期運用する危険性を、まさにこの監視システムが検知した例
- **リコンシリエーションが両方○**: PaperAccountの内部状態とイベント
  ログから再計算したポジション・モデル判断が完全に一致することを確認
- **監査の再構成**: ある1件の仮想約定について、対応するモデル推論
  （シグナル・信頼度）をログから遡って特定でき、「なぜこの日に
  この売買が起きたか」を説明できることを確認

## 監査可能性の実例

`demo.py`の最後で、1件の仮想約定を選び、対応するモデル推論イベントを
イベントログから逆引きして判断根拠を再構成する例を実行している。
すべての判断がJSONLとして記録されているため、後から任意の日・銘柄に
ついて「データ取得→検証→特徴量生成→モデル推論→仮想約定」の
一連の流れを再現できる。

## 限界・注意

- **自動発注・ブローカー接続は一切含まない**。実際のSMS/メール通知も
  行わず、`alerts.notify()`はログ記録＋標準出力のプレースホルダー
- モデルは学習後、再学習されない（静的）。ドリフト検知後にどう対応するか
  （再学習・停止・人間のレビュー）は本フレームワークのスコープ外
- ペーパートレードのコストモデルは単純なbps固定（スプレッド・
  マーケットインパクトは`../execution-algorithms/`の方がより精緻）
- リコンシリエーションは単純な数量突き合わせ。実務では約定単価・
  手数料・税処理まで含めた照合が必要
