# Sprint 1 振り返り（2026-07-03）

## 目的

部門エージェント（市場調査部など）を作り込む前に、会社OSの配線
（`/intake → /pm → /route → /portfolio`）が実際に機能するかを検証する。
特に、まだ実装されていない部門を指す案件をPMが正しく扱えるかを確認する。

## 結果

- `/intake` で「YouTube動画要約サービス」をPJ-001としてPortfolio Dashboardに
  登録し、市場調査部への引き渡しまで正常に動作した
- `/pm` はPJ-001を含む会社全体の状況（Today's Summary / Recommendation /
  Priority / CEOへの報告事項）を正しく生成した
- `/route PJ-001` は、市場調査部の成果物（調査レポート）が未提出であることを
  Exit Criteriaと照らして正しく検知し、存在しない部門の完了を勝手に仮定せず
  「Research待ち」として据え置いた
- `/portfolio` はPJ-001を含む全体像（案件数・優先度未評価・新規1件）を
  正しく集計した

## 成功

- OS配線（intake → pm → route → portfolio）が一通り成功した
- Exit Criteriaを尊重し、未実装の部門を待つ状態を正しく表現できた
  （前工程が未完了でも次に進んでしまう、というAIワークフローにありがちな
  誤動作が発生しなかった）

## 失敗・課題

- Research待ち（市場調査部が未実装のため、この先の検証ができない）
- 優先度・利益評価に必要なデータ（市場調査・評価結果）がまだ存在しないため、
  `/pm` の優先度表示・`/portfolio` の利益ランキングは実質空の状態にとどまった

## 改善

- Research部のMVPを作成し、実際に調査成果物を生成できるようにする
- Research部にはEvidence First（一次情報の証拠添付）とQuality Gate
  （完了条件のチェックリスト化）を組み込む

## 次回（Sprint 2）に向けたアクション

- Research部MVPを実装する（入力: アイデア／出力: 市場規模・競合5件・想定顧客・
  課題・収益モデル・リスク・参考URL）
- Research部にDefinition of Done（市場規模・競合5件以上・Evidence5件以上・
  TAM/SAM/SOM・収益モデル・リスク・PMレビュー）を設定する
- PJ-001を実際にResearch部MVPへ通し、`/route PJ-001` で事業評価部へ
  前進できるかを検証する
