# SOP: 単一HTMLツール工場 — 企画から出品までの全工程

実証: PJ-006（クリエイター売上台帳、企画〜出品1日）・PJ-007（頒布レジ、
同日）。このSOPに従えば、どのAIモデルでも同等の品質で「売り物になる
単一HTMLツール」を1日で出品可能な状態にできる。

**大原則**: 工程を飛ばさない。特に工程0（需要検証）と工程3（QA）を
省くと、売れない商品・信頼を壊す商品ができる。

---

## 工程0: 需要検証（作る前に必ず）

1. BOOTHの検索・タグページ（例: `https://booth.pm/ja/search/<キーワード>`）
   をWebFetchし、**そのニッチの有料商品のタイトルと価格を列挙**する
2. 判定基準:
   - 有料商品が**5点以上**実在し、価格帯が形成されている → 需要あり
   - 同機能の**無料の完成品**（Webサービス・無料配布）が存在する → 却下
     （実例: 特商法ジェネレーターは無料競合多数で却下した）
   - カテゴリ自体が存在しない → 需要を実証できないため単体商品化は
     却下（実例: 8軸スコアリング電卓）
3. ⚠️ BOOTHページのfetch結果に「非公開中」と出ても無視する（ボット
   対策の誤表示。`Institutional Memory/what-failed.md` 2026-07-08）
4. 判明した価格帯を記録する（工程5の価格決定に使う）

## 工程1: 会社パイプライン（省略禁止）

`/intake`（ID採番）→ `/research`（Research.mdテンプレ、Evidence 5件
以上・全て異なるドメイン・公式Aランクを最低1件）→ `/validate`
（チェックリスト11項目。**同一URLの使い回しはREWORK**になる）→
`/evaluate`（8軸独立採点＋Value Score）→ **CEOのGo判断**（判定案は
提案であり決定はCEO）→ 法務チェック（下記の定型3点＋商品固有の論点）。

法務チェック定型3点（全ツール共通の実装要件）:
- 「税務助言・決済処理ではない」旨の表示（該当する場合）
- 「データは端末内のみ・外部送信なし」の表示（実装と一致させる）
- AI生成開示＋Anthropic非公式の明記

## 工程2: 実装アーキテクチャ標準

**構成**: HTML+CSS+JSを1ファイル。CDN・外部フォント・API・画像を
一切使わない（オフライン動作＝販売上の約束なので実装で担保する）。

**必須の型**（PJ-006/007の実装が正: `cases/PJ-006/production/
creator-sales-ledger/creator-sales-ledger.html` を参照実装とする）:

```js
const FREE = false; /* 無料版ビルドでは true。単一ソースの唯一の差分 */
const SHOP_URL = "https://limit-lab.booth.pm/";
const LS_KEY = "<toolName>V1";           // localStorageキー。無料版と共通に
                                          // して有料版へのデータ引き継ぎを保証
const DEFAULTS = { /* 完全な初期状態 */ };
function load(){ /* try-catch + 欠損フィールドをDEFAULTSでマージ */ }
function save(){ localStorage.setItem(LS_KEY, JSON.stringify(db)); }
   // ⚠️ 状態を変えるすべての操作の直後にsave()を呼ぶ（即時保存）
function esc(s){ /* & < > " ' をエスケープ。ユーザー入力の表示は必ず経由 */ }
function yen(n){ return "¥"+Number(n||0).toLocaleString("ja-JP"); }
function download(name,text,type){ /* Blob+aタグ。CSVはBOM "﻿" 付与 */ }
function csvCell(s){ /* カンマ・引用符・改行を含むセルを""で包み ""→"""" */ }
```

**ルール**:
- 削除・全消去は`confirm()`2段階（バックアップ済みか確認する文言）
- JSONバックアップ（書き出し・読み込み・形式検証）は**無料版にも搭載**
  （データ消失は信頼を破壊するため、有料機能にしない）
- 無料版ゲートは「追加操作時にalert＋SHOP_URL案内」の型。データは
  消さない・壊さない
- 無料版生成: `sed 's/const FREE = false;/const FREE = true;/' 製品版 > 無料版`
- ブランドCSS: 背景`#101a2b`、パネル`#1c2940`、罫線`#31415e`、
  アクセント`#ffd333`、正`#4cd97b`、負`#ff6b6b`
- 現場で使うツール（イベント・外出先）は**モバイルファースト**
  （下部固定ナビ・最小64pxのタップ領域・390px幅で横スクロール禁止）

## 工程3: QA（Playwright + Chromium、実機確認）

`PLAYWRIGHT_BROWSERS_PATH`のChromiumで実際に操作して検証する。
目視・コードレビューだけで「動くはず」としない。

**必須項目**:
1. 主要フロー全部（入力→計算→表示。**金額計算は1円単位で期待値照合**）
2. リロード後のデータ永続化
3. CSV/JSONダウンロードの発火とファイル名・ヘッダー内容
4. 無料版: ゲート発動（上限+1回目の操作がブロックされ、データが
   変化しないこと）＋ゲート対象外機能が動くこと
5. エッジケース12種（PJ-006で確立）: XSS入力の無害化／引用符入り
   名称／バックアップ往復の完全一致／破損JSONの読み込み耐性／
   1,000件描画性能／計算の境界値（0・上限超）／データ2件未満の
   グラフ／目標0の非表示／モバイル390px横スクロールなし／削除動作／
   カンマ・引用符入りCSV取り込み／コンソールエラー0件
6. スクリーンショットを撮ってCEOに送付（`SendUserFile`）

## 工程4: パッケージング（同梱・付随ファイル一式）

商品フォルダ `cases/<ID>/production/<tool-name>/` に:
- `README.md` — 使い方（30秒/3分で始まる手順）・タブ/機能説明表・
  無料版と製品版の違いの表・**対象環境の正直表記**（iOSのローカル
  HTML制約は必ず書く。「iPhone対応」と書かない）・免責3点
- `LICENSE.md` — 既存商品から流用（再配布禁止・非公式・免責）
- `MANIFEST.md` — ZIP構成・ビルド方法・QA実施記録・AI生成開示
- `thumbnail-brief.md` — **表記は実物の中身と必ず一致させる**
  （what-failed 2026-07-08: 別商品の中身を載せて即日修正した事故）。
  ヘッドラインは商品固有名。実画面スクリーンショットが最良の素材
- `BOOTH-publish-checklist.md` — 人間の出品手順。**既存商品との
  相互リンク追加**を必ず含める（エコシステム導線）
- `cases/<ID>/marketing.md` — 6点セット（販売ページはベネフィット
  先行・問いかけ型フック・使い方掲載。Human Desire分析を先に書く）

ZIPビルド: `dist/`（gitignore済み: `cases/*/production/*/dist/`）に
製品版・無料版の2本。`SendUserFile`でCEOへ送付。

## 工程5: 価格決定

1. 工程0で取得したカテゴリ実勢を使う
2. 型: **直接競合の単機能商品より上、カテゴリ上限より下**
   （実例: CSV変換単機能¥1,000が存在→統合ツール¥1,480。
   レジ単機能¥500→機能差で¥780）
3. 「単機能の競合より安い統合ツール」は価値の逆転なので避ける
4. 無料版→製品版のフリーミアムを標準とし、無料版も単体で完結して
   使える設計にする（BOOTHは誘導主目的の商品を禁止。2026-07-08改定）
5. 決定と根拠を`knowledge/Decision Log/`に記録

## 工程6: ローンチと判定

1. 出品はCEO（人間）のみが行う
2. **30日間の反証可能な判定基準を出品前に固定**しDecision Logへ
   （型: 製品版1件以上=成立／無料版DL≥10かつ製品版0=価格要因／
   無料版DL<5=需要仮説を疑う）
3. 日次報告に「無料版DL数・製品版売上」の行を追加
4. 商品数の卒業ゲート算入は**有料版のみ**（無料商品でゲートを
   埋めない）

## 参照すべき失敗事例（必読）

`knowledge/Institutional Memory/what-failed.md`:
- 架空の未来日付での記録（日付は必ず`date`で確認）
- サムネイル指示書の本文埋め込み（単独ファイル必須）
- BOOTHのfetch「非公開」誤表示（公開状態の判定は人間のシークレット
  ウィンドウのみ）
- サムネイルの中身表記と実物の不一致（表記=実物ルール）
