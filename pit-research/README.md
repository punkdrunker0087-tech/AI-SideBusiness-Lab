# Point-in-Time Research（Project B）― `aqm-strategy/`の続きではない

> 研究・学習用。発注機能は持たない。

## これは何のプロジェクトか

`../aqm-strategy/`（"Price-only Research v1.0"、2026-07-23に凍結）で、
「価格・出来高だけから独立したアルファ・情報軸を探索できるか」という
問いを検証した。ユニバース拡大・パラメータ探索・DSR/PBO・15年複数
レジーム検証・FX/金利ベータ・ループ学習・相関/PCA/クラスタリング・
第3軸候補（短期リバーサル・IVOL・歪度・共動性）まで一通り試した結果、
**価格・出来高だけでは独立した第3の情報軸を確認できなかった**、という
結論に達した。

このプロジェクトは、その**続きではない**。問い自体が異なる:

| | Project A（`aqm-strategy/`） | Project B（このプロジェクト） |
|---|---|---|
| 問い | 価格・出来高だけで独立したアルファを作れるか | Point-in-Time情報を加えると本当に新しい情報軸が得られるか |
| データ | Yahoo Finance（無料・現在スナップショットのみ） | J-Quants（無料プラン・Point-in-Time・過去2年） |
| 結論 | 凍結済み（見つからなかった） | 検証中 |

## データソースの制約（重要・先に理解しておくこと）

J-Quants無料プランは**過去2年分・12週間の開示遅延**という制約がある
（AQM-01シリーズの調査で確認済み）。これは:

- Project Aで使った**15年間の複数レジーム検証はこのデータだけでは
  組めない**（COVID禍・Abenomics相場等は範囲外）
- 直近2年という短い期間での検証になるため、DSR/PBOの多重検定補正が
  Project A以上に重要になる（サンプルが少ないほど偶然の当たりが
  出やすい）
- 「Point-in-Time情報を使っても第3軸が見つからなかった」という結果が
  出た場合、それが「情報源自体に価値がない」のか「検証期間が短すぎた」
  のか、Project Aほど明確に切り分けられない可能性がある、という限界を
  先に認識しておく

## 検証の進め方（Project Aの反省を踏まえた順序）

Project Aでは、Reversal・IVOLという「実は既存ファクターと数学的に近い」
候補を先に試してしまい、後から歪度・共動性という真に独立な候補を
追加検証する、という手戻りがあった。Project Bではこの反省を活かし、
**いきなり大量のファクターを作らず、1つの候補について次の順序を必ず
守る**:

1. **候補を1つ作る**: 例）EPS修正率（前四半期予想からの変化率）、
   ROEのPoint-in-Time値、など。
2. **①新規性ゲート**: Project Aで確立した2テーマ（テーマA=マクロ/
   リスクオン軸、テーマB=価格テクニカル軸）との相関・クラスタリングを
   確認する。相関が高ければ「新しい情報ではない」として、それ以上の
   検証をしない。
3. **②単独DSR/PBOゲート**: 新規性を確認できた候補だけ、単独のL/S
   バックテストでDSR・PBOを計算する。
4. **③限界貢献度ゲート**: ②を通過した候補だけ、既存ファクタープール
   に加えた際の実際の改善（ループ学習・固定配分どちらでも）を測る。
5. 3段階すべてを通過した候補だけを採用する。

## 構成

| ファイル | 内容 |
|---|---|
| `jquants_client.py` | J-Quants API(V2)クライアント。`x-api-key`認証、`/fins/summary`（決算短信サマリー）・`/equities/bars/daily`（株価四本値）取得 |
| `eps_revision_factor.py` | EPS修正率のPoint-in-Timeパネル構築（`DiscDate`基準、年度切り替わりは除外） |
| `eps_revision_gate_test.py` | EPS修正率の3段階ゲート検証（①相関→②DSR/PBO→③限界貢献度） |

## セットアップ

J-Quants APIはV2（2025-12-22以降に登録したアカウントはV2のみ利用可）。
認証はAPIキー方式（`x-api-key`ヘッダー）のみで、Googleアカウントでの
登録でも問題なく使える。ベースURLは実地の接続テストで
`https://api.jquants.com/v2`と確認済み（`api.jquants-pro.com`ではない
点に注意）。

```bash
cd pit-research
pip install -r requirements.txt

# マイページ「API Keys」から発行したキーを設定
export JQUANTS_API_KEY="発行したAPIキー"

# 接続確認
python jquants_client.py
```

### 1つ目の候補「EPS修正率」の検証結果（`eps_revision_factor.py`・`eps_revision_gate_test.py`）

決算短信の`NxFEPS`（通期開示時の翌期予想）と`FEPS`（四半期開示時の
当期予想更新）を開示日順に連結し、同一年度内の修正率を計算（年度
切り替わりの回はNaNとして除外）。`DiscDate`の翌日から値が有効になる
Point-in-Timeパネルを構築し、3段階ゲートにかけた:

- **①新規性ゲート**: 既存5ファクター（Momentum・LowVol・Liquidity・
  FX・Rate）との相関は最大0.478（Liquidityと）。閾値0.5をかろうじて
  下回り合格としたが、**かなり際どい水準**であることは明記する。
- **②単独DSR/PBOゲート**: top_q×rebalanceの正真正銘6通りのグリッドで
  最良Sharpe+0.716、**DSR=0.736だがPBO=0.750**。DSR単独では合格に
  見えるが、PBOが極めて高く過学習の強い懸念があるため、**DSR>0.5かつ
  PBO<0.3の両方を要求する基準で不合格**と判定した。
- **③限界貢献度ゲート**: ②で不合格のため本来は実施しないが、参考までに
  実施したところ、既存5ファクター合成へEPS修正率を加えると
  Sharpe+1.11→+0.99・総リターン+20.4%→+17.2%と**むしろ悪化**した。
  ②の不合格判定と整合的な結果。

⚠️ **実装中に発見したバグ**: 当初のグリッドサーチで、EPS修正率には
FX/Rateのような自然な「ローリング窓」パラメータがないため、便宜的に
同じ設定を3回複製しただけの偽のグリッド（6通りと称して実質2通り）を
組んでしまい、DSRを不当に押し上げていた。top_q×rebalanceの正真正銘の
グリッドに差し替えて修正した。

**結論**: EPS修正率（1つ目のPoint-in-Time候補）は、新規性は際どく合格
したが、DSR/PBOの統計的評価で不合格。2年間という短い検証期間（Project
Aの15年より遥かに短い）ではサンプルが少なく、DSRが基準を満たしても
PBOが高く出やすいことも一因と考えられる。次の候補（ROEのPoint-in-Time
値など）に進む。
