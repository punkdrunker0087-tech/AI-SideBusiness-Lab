# アルゴリズム取引 コンプライアンス＆ガバナンス・フレームワーク

「誰が・いつ・何を変更したかを追跡できること」「すべての判断と例外を
説明できること」「障害時に安全側へ移行できること」「規制・監査に耐えうる
証跡を維持すること」を中核に据えた、統制・追跡・承認ワークフローの実装。

> **自動発注・ブローカー接続は一切含まない。** 取引前チェック・限度管理・
> 市場公正性の検知はすべて「判定・記録」のみで、実際の注文送信は行わない
> （このリポジトリ全体の方針と同じ）。

## 12項目と実装の対応

| # | 項目 | 実装 |
|---|---|---|
| 1 | ガバナンス構造 | `governance.py`（三線防衛モデル・職務分掌の強制） |
| 2 | 規制インベントリ | `regulatory_inventory.py`（管理台帳・例示のみ） |
| 3 | 取引前リスクコントロール | `pretrade_controls.py`（数量・価格・ポジション・システム健全性） |
| 4 | ポジション限度 | `position_limits.py`（個別/セクター/資産クラス/全体、ソフト/ハード） |
| 5 | 市場公正性 | `market_integrity.py`（自己約定・過剰キャンセル・引け集中の検知） |
| 6 | 最良執行 | `best_execution.py`（執行記録・スリッページ） |
| 7 | 記録保持 | `audit_log.py`（全モジュール共通の監査ログ） |
| 8 | 変更管理 | `change_management.py`（状態機械で工程飛ばしを禁止） |
| 9 | インシデント対応 | `incident_response.py`（検知〜再発防止のライフサイクル） |
| 10 | レビュー計画 | `review_calendar.py`（日次〜年次の期限管理） |
| 11 | 税務・会計 | `tax_records.py`（記録構造のみ・税務助言ではない） |
| 12 | KPI | `kpi_dashboard.py`（監査ログから自動集計） |

## セットアップ & 実行

```bash
cd compliance-governance
pip install -r requirements.txt
python demo.py
```

## 実測結果（合成シナリオでの検証）

全12項目を実際に動かし、想定した異常系がすべて正しく検知されることを確認した。

| シナリオ | 結果 |
|---|---|
| 開発者が自分の変更を自分で承認しようとする | ⚠️ 職務分掌違反として拒否 |
| 指値が市場価格から10倍乖離した誤発注 | 拒否（`price_deviation`） |
| 上限を大きく超える数量の発注 | 拒否（`quantity`・`notional`） |
| 個別銘柄・セクターの残高がソフトリミット超過 | 🟡SOFT警告（ハードリミットは未超過と正しく区別） |
| 同一口座が同時刻に同銘柄を買い・売り | 自己約定の疑いとしてフラグ |
| 1口座がキャンセル率90%で発注 | 過剰キャンセル比率としてフラグ |
| 変更管理でテスト工程を飛ばして承認しようとする | ⚠️ 状態機械が不正遷移として拒否 |
| 正規の変更フロー（テスト→レビュー→承認→反映→事後確認） | 完走・全工程を監査ログに記録 |
| インシデント（検知→初動→エスカレーション→原因分析→復旧→再発防止） | 全ライフサイクルを記録、初動までの時間を計測 |
| 週次レビューが期限を3日超過 | レビューカレンダーが正しく検出 |

KPIダッシュボードは監査ログを集計するだけで、リミット超過件数・取引前
チェック拒否件数・市場公正性フラグ件数・変更管理のロールバック率・
インシデント件数を自動算出する。

## 設計上の要点

- **記録保持はすべて共通の`audit_log.py`を経由する。** 市場データ・
  シグナル・リスク判定・注文・約定・ポジション・モデルバージョン・
  設定値・システムログ・監査ログを一貫した形式（JSONL・actor付き）で残す
- **職務分掌はコードで強制する。** `governance.verify_segregation()`は
  提案者と承認者が同一人物、または両者とも第一線の場合に例外を送出する
- **変更管理は状態機械。** draft→tested→reviewed→approved→deployed→
  post_reviewed の順序を強制し、工程を飛ばした遷移は`InvalidTransitionError`
  で拒否される
- **市場公正性の検知は自動ブロックしない。** 誤検知のリスクがあるため、
  フラグを立てて人によるレビューに委ねる設計

## 限界・注意（必ず読む）

- **規制インベントリの内容は構造の例示であり、法的助言ではない。** 実際の
  適用法令・管理方法は市場・証券会社・投資主体・法域によって異なるため、
  必ずコンプライアンス部門・弁護士に確認すること
- **税務記録も同様に税務助言ではない。** 実際の申告前には税理士等の専門家に
  確認すること
- 市場公正性の検知ロジックは簡易なヒューリスティック（実際のサーベイランス
  システムはより高度な統計的検定・機械学習を用いることが多い）
- インシデント対応の連絡先（`ESCALATION_CONTACTS`）は仮の値。実際の運用では
  組織の緊急連絡網に置き換える必要がある
- 発注機能なし
