# 市場データ基盤

一般的な市場データ基盤のレイヤー構成（取得→検証・正規化→保存→特徴量生成→
API→研究・監視）を実データで実装した学習用フレームワーク。

> 研究・学習用。ティック/分足は設計のみ（同一アーキテクチャを拡張すれば
> 対応可能）、実データ検証は日足のみ。発注機能は持たない。

## レイヤー構成と実装の対応

```
データ提供元 → データ取得 → 検証・正規化 → 保存(生/加工済み/メタデータ)
  → コーポレートアクション → 特徴量生成(フィーチャーストア) → API → 研究・監視
```

| レイヤー | 実装 |
|---|---|
| データ取得 | `ingestion.py`（提供元・取得時刻・粒度をメタデータとして記録） |
| 検証・正規化 | `validation.py`（欠損・重複・外れ値・順序・OHLC整合性・型） |
| 保存 | `storage.py`（raw/processed/metadataを分離、更新履歴を追記） |
| コーポレートアクション | `corporate_actions.py`（分割/配当らしき乖離を検知） |
| フィーチャーストア | `feature_store.py`（元データのハッシュで再現性を検証） |
| API・利用インターフェース | `api.py`（権限・キャッシュ・監査ログ） |
| スケジューリング | `scheduler.py`（データ種別ごとの更新要否判定） |

## セットアップ & 実行

```bash
cd market-data-platform
pip install -r requirements.txt
python demo.py
```

## 実測結果（1306.T・7203.T・5年）

### コーポレートアクション検知が実データで機能

- **1306.T（TOPIX ETF）**: 毎年7月前後に5件の配当らしきイベントを検出
  （2022-07-08, 2023-07-10, 2024-07-10, 2025-07-10, 2026-07-10）。
  TOPIX連動ETFの分配時期と整合する規則的なパターン
- **7203.T（トヨタ）**: 3月末・9月末付近に配当らしきイベントを検出
  （決算期末3月・中間9月の配当と整合）

### 検証レイヤーが実際のデータ品質問題を検出

1306.Tで「±50%超の日次変化」を2件検出。中身を調べると、**2026-03-30〜
03-31に価格が約375円→37円へ2日間誤って落ち込み、04-01に383円へ復帰する
Yahoo側のデータグリッチ**だった（`../aqm-strategy/panel.py`で対処した
ものと同一の実データ上の問題）。

重要なのは、このグリッチが**コーポレートアクションとして誤分類されず**
（配当・分割どちらの乖離パターンにも一致しない）、かつ`normalize()`が
**これを黙って自動修復もしない**（設計方針: 「原因確認前に下流で無条件に
使われないように」）という、設計通りの挙動を実データで確認できたこと。
検証レポートに警告として残り、人間・下流処理の判断に委ねられる。

### フィーチャーストアの再現性検証

`momentum_20d`特徴量を登録し、同一ソースデータでは`reproducible=True`、
末尾5行を落としたデータでは`reproducible=False`（ハッシュ不一致）と
なることを確認。改変を検知できる設計が機能している。

### API層: 権限・キャッシュ・監査ログ

- `research`/`backtest`は許可され、`unknown_service`は`PermissionError_`で
  正しく拒否
- 2回目の呼び出しはキャッシュヒットとしてディスクアクセスを回避
- 監査ログに呼び出し元・アクション・タイムスタンプが記録される

## 保存設計の詳細

```
data/raw/{symbol}/{granularity}.csv        取得したそのままのOHLCV
data/processed/{symbol}/{granularity}.csv  検証・正規化済みOHLCV
data/metadata/{symbol}.json                提供元・取得時刻・更新履歴・
                                            検出したコーポレートアクション
data/features/{name}/{symbol}_{version}.csv
data/features/registry.json                特徴量のハッシュ・計算方法・
                                            バージョン
```

粒度（ティック/分足/日足）ごとにディレクトリ・ファイルを分ける設計に
しており、更新頻度・保持期間をそれぞれ独立に管理できる。

## データソースの扱い

本実装ではYahoo Finance公開APIのみを使うが、設計上は複数種のデータソース
（財務・企業イベント・マクロ・ニュース・代替データ）を扱うことを想定し、
`ingestion.FetchResult`に**提供元・取得時刻・粒度・ライセンス注記**を
必ず記録する構造にしている。データ種別ごとに更新頻度・品質・ライセンス
条件が異なるため、この記録が後の監査・再検証の土台になる。

## 限界・注意

- ティック/分足データは無料経路がなく設計のみ（`scheduler.UpdateFrequency.
  TICK`として区分は用意）
- コーポレートアクション検知は「分割らしき急変」「配当らしき小さな乖離」の
  2分類に限定。合併・スピンオフ・上場廃止は価格系列だけからの自動判別が
  難しく、"unclassified"（要確認）として保守的に扱う設計だが、本デモの
  期間には該当イベントが含まれていなかった
- 監査ログ・権限管理は単純な実装（実運用にはユーザー/サービスごとの
  ロール管理への発展が必要）
- 発注機能なし
