# マーケットメイキング ―― 基本構造の学習フレームワーク

**方向性を当てるのではなく、在庫を一方向に偏らせないこと**を中心に据えた
マーケットメイキングの学習用実装。Avellaneda-Stoikov (2008) の在庫リスク・
クォートモデルを軸に、在庫管理・マイクロストラクチャー指標・リスク管理・
多面的な評価指標を実装する。

> 研究・学習用。実発注機能は持たない。**リテールが実市場でマーケットメイクを
> 行うのは執行速度で機関投資家に構造的に劣後し現実的でない**（下記「限界」参照）。
> 目的は「市場がどう価格を形成するか」を理解すること。

## 3つの収益・リスク要因と実装対応

| 要因 | 内容 | 実装 |
|---|---|---|
| **スプレッド収益** | 買値と売値の差から得る収益 | `quoting.py`（最適スプレッド）・`metrics.py`（PnL分解） |
| **在庫リスク** | 保有ポジションの価格変動損益 | `inventory.py`（在庫感応度・制限・ヘッジ） |
| **逆選択リスク** | 情報優位者との取引で不利に約定するリスク | `metrics.py`（マークアウト分析で定量化） |

## 構成

| ファイル | 内容 |
|---|---|
| `quoting.py` | Avellaneda-Stoikovクォートモデル（留保価格・最適スプレッド） |
| `inventory.py` | 在庫の価格感応度・制限判定・部分ヘッジ計算 |
| `simulation.py` | イベント駆動シミュレーション（GBM仲値＋ポアソン到着） |
| `metrics.py` | 約定率・保有時間・在庫変動・PnL分解・リスク調整後リターン |
| `microstructure.py` | 実データからの代理指標（Rollスプレッド・Amihud非流動性） |
| `demo.py` | 全体を通したデモ |

## 在庫管理の考え方（核心）

Avellaneda-Stoikovモデルは、仲値ではなく**在庫を考慮した留保価格**を中心にクォートする。

```
reservation_price = mid − inventory・γ・σ²・(T−t)
optimal_spread    = γ・σ²・(T−t) + (2/γ)・ln(1+γ/κ)
```

在庫が買い持ちに傾くほど留保価格を下げて売れやすくし、売り持ちに傾くほど
上げて買われやすくする ―― これが「方向を当てる」のではなく「在庫を中立に
保つ」設計そのもの。

### 実測: γ（リスク回避度）のトレードオフ

| γ | 約定数 | 在庫std | 在庫最大絶対値 | Sharpe/step | 累積PnL |
|---|---|---|---|---|---|
| 0.01 | 2650 | 0.99 | 6.0 | 0.48 | 1841 |
| 0.05 | 837 | 0.44 | 4.0 | 0.18 | 571 |
| 0.10 | 454 | 0.31 | 4.0 | 0.12 | 307 |
| 0.30 | 172 | 0.16 | 3.0 | 0.07 | 107 |
| 0.60 | 72 | 0.10 | 2.0 | 0.04 | 45 |

γを上げる（在庫を嫌う）ほどスプレッドが広がって約定数が減り、在庫は安定するが
収益機会も減る。**「安全運転」と「スプレッドを稼ぐ」は表裏一体**で、これが
マーケットメイクの中心的なチューニング問題。

## マイクロストラクチャーで観察される要素

| 要素 | 本フレームワークでの扱い |
|---|---|
| 板の厚さ・注文フローの偏り・キャンセル率 | シミュレーション(`simulation.py`)でのみ厳密に扱える |
| スプレッド | 実データ: Rollの実効スプレッド推定量（`microstructure.py`） |
| 出来高・流動性 | 実データ: Amihud非流動性・実現ボラティリティ |

⚠️ **重要な限界**: 無料の日次OHLCVには板情報（気配・注文の内訳）がないため、
「板の厚さ」「注文キャンセル率」「注文フローの偏り」を実データから厳密に
測ることはできない。実測ではRoll推定量（トヨタ48円、JR東日本47円）と
Amihud非流動性（トヨタ2.0e-7 < JR東日本1.2e-6、トヨタの方が流動的）を
代理指標として使用。ティックサイズ・時間帯パターンは日足では原理的に扱えない。

## リスク管理・評価指標

- **在庫制限**: `InventoryLimits`（soft/hard）でクォート可否を判定。
  実測でハードリミット(5)超過を正しく抑制することを確認
- **ヘッジ**: `hedge_size()` で目標在庫への部分ヘッジ量を計算
- **PnL分解**: マークアウト分析でスプレッド収益と逆選択コストを分離。
  情報優位者のいない合成市場では逆選択コストがほぼゼロ（実測0.01）に
  なることを確認 ―― これはロジックの正しさの検証でもある（実市場では
  情報優位者の存在により逆選択コストが正に効き、収益を削る）
- **多面的評価**: 約定率・平均保有時間・在庫変動・リスク調整後リターンを
  `metrics.full_report()` で一括出力

## 学習に役立つ研究テーマ（参照）

- Avellaneda, M. & Stoikov, S. (2008) "High-frequency trading in a limit order book"
- Roll, R. (1984) "A Simple Implicit Measure of the Bid-Ask Spread"
- Amihud, Y. (2002) "Illiquidity and Stock Returns"
- 市場マイクロストラクチャー全般: 指値/成行注文の相互作用、流動性供給者・
  需要者の役割、オーダーブックのダイナミクス、価格発見メカニズム

## 限界・注意（必ず読む）

- **実データではなく合成市場（GBM＋ポアソン到着）での設計検証**。実際の
  収益性は実市場の板・手数料体系・約定順位・レイテンシに強く依存し、ここでは扱わない
- **リテールがマーケットメイクで実際に稼ぐのは構造的に困難**。プロのMMは
  コロケーション・専用回線・優先的な板アクセスで速度優位を持ち、リテールの
  執行速度では逆選択（新しい情報が反映された後にしか約定できない）を
  避けにくい。本フレームワークは「なぜそうなるか」を理解するための学習教材
- 板情報がないため、マイクロストラクチャー指標の一部は代理指標に留まる
- 発注機能なし。実運用は別の正規手段で行う
