# 執行アルゴリズム研究フレームワーク

執行アルゴリズムの役割は「市場を予測する」ことではなく、決められた売買を
できるだけ効率よく実行すること。**市場インパクトを抑える・スリッページを
抑える・約定の確実性を高める**の3目的のバランスを、TWAP・VWAP・アイスバーグ・
スマート注文ルーティングで比較し、TCA（Transaction Cost Analysis）で
評価する。

> 研究・学習用。板情報がないため、合成市場（U字型出来高・GBM仲値・平方根法則
> のマーケットインパクト）での検証（`../market-making/`と同じ設計方針）。
> 実発注機能なし。

## 構成

| ファイル | 内容 |
|---|---|
| `market_sim.py` | 合成市場（U字型出来高プロファイル・GBM仲値・出来高連動スプレッド） |
| `impact_model.py` | 平方根法則によるマーケットインパクト（一時的・恒久的） |
| `engine.py` | スケジュール（各刻みの発注量）を合成市場に対して執行するエンジン |
| `twap.py` | 時間的に均等分散 |
| `vwap.py` | 出来高分布に比例、予測出来高との乖離も扱う |
| `iceberg.py` | 表示数量の制限による情報漏洩軽減と約定率のトレードオフ |
| `smart_router.py` | 複数venue（取引所・PTS）へのコスト最小化ルーティング |
| `tca.py` | Implementation Shortfallの要素分解・VWAPベンチマーク比較 |
| `monitoring.py` | 複数日にわたる継続的モニタリングとアラート |
| `demo.py` | 全体を通した実演 |

## セットアップ & 実行

```bash
cd execution-algorithms
pip install -r requirements.txt
python demo.py
```

## 実測結果と得られた教訓

### 1. TWAP vs VWAP vs アイスバーグ

| 手法 | 約定率 | IS(bps) | インパクトコスト(bps) | 対VWAP(bps) |
|---|---|---|---|---|
| TWAP | 98.2% | +34.0 | 8.8 | +16.6 |
| VWAP(予測出来高) | 100.0% | +29.1 | 8.1 | +11.7 |
| VWAP(完全な出来高・理論上限) | 100.0% | +27.9 | 8.0 | +10.5 |
| アイスバーグ(表示2%) | 100.0% | +28.8 | **3.2** | +11.4 |

- **VWAPがTWAPよりインパクトコストが低い**（8.0-8.1 vs 8.8bps）: 出来高が
  薄い時間帯への発注を避けた効果
- **アイスバーグはインパクトコストが大幅に低い**（3.2bps）: 表示数量を絞る
  ことで恒久的インパクト（情報漏洩）を軽減。ただし刻み別の平均約定率は
  85.0%に低下しており、**インパクト軽減と約定の確実性はトレードオフ**
  であることが数字で確認できる

### 2. VWAPの出来高予測誤差と執行品質の関係

| 予測ノイズ | 分布の乖離(tracking error) | IS(bps) | 約定率 |
|---|---|---|---|
| 0.00（完全予測） | 0.000 | +27.9 | 100.0% |
| 0.15 | 0.059 | +29.1 | 100.0% |
| 0.40（大きく外れる） | 0.155 | +31.1 | 98.5% |

予測出来高が実際の分布から乖離するほど、ISが単調に悪化し約定率も低下する
――「VWAPは将来の出来高予測に依存し、乖離すれば追随が崩れる」という
研究上の論点を、実測で単調な関係として確認できた。

### 3. スマート注文ルーティング

流動性シェアに比例するだけの素朴な配分（期待コスト3.35bps）に対し、
手数料・スプレッド・レイテンシから算出したコストスコアで安い先から
優先配分するウォーターフォール方式（3.07bps）は**+0.28bpsの改善**。
改善幅は小さいが、venue間のコスト差が小さい設定でもルーティングが
一貫して同方向に効くことを確認。

### 4. TCAの要素分解で見つけた重要な教訓（実装中に発見・修正）

実装当初、Implementation Shortfall（到着価格基準、+34.0bps）と、
その内訳（スプレッド+インパクト+遅延+機会+手数料 = 15.2bps）が
**一致しないというバグ**があった。原因は、執行期間中に市場そのものが
ランダムに動く「市場ドリフト」の要素を分解に含めていなかったこと。

修正して`market_drift_bps`（残差＝執行スキルと無関係な市場の値動き）を
明示的な項目として追加したところ、20日分のモニタリングで次の関係が
判明した:

| 指標 | 平均 | 標準偏差 |
|---|---|---|
| IS（到着価格基準） | +37.2bps | **91.6** |
| 対VWAPベンチマーク | +12.6bps | **7.8** |
| 市場ドリフト（残差） | +19.3bps | 88.7 |

**IS単体の標準偏差(91.6)は、ほぼ市場ドリフトの標準偏差(88.7)で説明される。**
つまり到着価格基準のISは執行期間中の市場のランダムな値動きに支配され、
日々の執行スキルの変化を判定するには**極めてノイジー**。一方、**VWAP
ベンチマークとの比較（標準偏差7.8）の方がはるかに安定しており、執行
アルゴリズムの実力を監視するにはこちらが実務的**、という発見は実装
過程で偶然見つかったが、TCA研究で広く知られる知見と整合する。
`monitoring.py`のアラートも、この理由からIS単体でなく対VWAPコストを
主指標にするよう設計している。

## 執行品質評価の考え方（TCAの要素分解）

```
Implementation Shortfall = スプレッドコスト + マーケットインパクトコスト
                          + 遅延コスト + 機会コスト + 手数料 + 市場ドリフト
```

- **スプレッドコスト**: 支払った半スプレッドの出来高加重平均
- **マーケットインパクトコスト**: 約定価格と「自分のインパクトを除いた
  市場価格」の差（一時的インパクトに起因）
- **遅延コスト**: 執行開始（最初の約定）までに市場が動いた分
- **機会コスト**: 未達数量について、セッション終値までの価格変化を
  機会損失として計上
- **市場ドリフト**: 上記どれにも起因しない、執行期間中の市場そのものの
  値動き（執行スキルとは無関係の残差）

## 限界・注意

- 板情報がないため合成市場での検証。実市場の絶対値は板構造・参加者構成に
  強く依存し、ここでは「手法間の相対比較・トレードオフの理解」が目的
- マーケットインパクトは平方根法則という単純化されたモデル
  （Almgren, Thum, Hauptmann, Li (2005)）。実際のインパクトは銘柄・
  市場構造でパラメータが異なる
- スマートルーティングは単純なウォーターフォール・ヒューリスティックで
  あり、真の最適配分（動的計画法等）ではない
- 発注機能なし
