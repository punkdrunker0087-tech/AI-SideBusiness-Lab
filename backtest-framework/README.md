# 教育用バックテストフレームワーク

**「コードを書く」より「検証設計を正しくする」**を学ぶための、段階的な
サンプル実装集。定量ファンドが研究を進める7段階の枠組みに沿って、
データ品質チェック・ウォークフォワード・モンテカルロ・統計検定を実装する。

> 本格的な自動売買システムとは区別した**研究・学習用**。発注機能は持たない。
> 「バックテストでは勝っていたのに実運用で通用しない」典型的失敗を、
> なぜ起きるか・どう検出するかを手を動かして理解することが目的。

## 定量研究の7段階と、本リポジトリの対応

| 段階 | 内容 | 実装 |
|---|---|---|
| **1. データ監査** | 欠損・重複・異常値・分割/配当調整・タイムスタンプ整合性 | `data_audit.py` |
| **2. バイアス除去** | 先読み・生存者・データスヌーピング・選択バイアス | 各所＋下記解説 |
| **3. ウォークフォワード** | 学習/検証を分離、定期再学習で未知データ評価 | `../rakuten-backtest/walkforward.py` |
| **4. コストモデル** | 手数料・スプレッド・スリッページ・市場インパクト・約定確率 | `engines.py`＋下記解説 |
| **5. ロバスト性評価** | パラメータ感度・モンテカルロ・ブートストラップ・ストレス | `robustness.py` |
| **6. 統計評価** | Sharpe/Sortino/IR＋**DSR・White's Reality Check・PBO** | `stats_eval.py` |
| **7. ペーパートレード** | 実資金なしのライブ検証、実運用との差異測定 | （次の段階・未実装） |

`demo.py` が Stage1→6→5 を実データ（日経225）で通しで実演する。

## セットアップ & 実行

```bash
cd backtest-framework
pip install -r requirements.txt
python demo.py            # 日経225で全段階デモ
python demo.py 7203.T     # 銘柄を変える
```

## Stage 2: バイアス除去（最重要の考え方）

| バイアス | 何が起きるか | 対策（本実装での扱い） |
|---|---|---|
| **先読み(look-ahead)** | 未来の情報で過去の判断をする | シグナルを1日ずらす（`positions.shift(1)`）。イベント駆動型は当日確定情報のみ使用 |
| **生存者(survivorship)** | 退場した銘柄を除くと成績が過大評価 | 上場廃止銘柄を含むデータを使う（無料データでは困難＝限界として明記） |
| **データスヌーピング** | 多数試して偶然の勝者を"発見" | DSR・PBO・Reality Checkで補正（Stage6） |
| **選択(selection)** | 良い結果だけ報告する | 試行回数を数え、DSRの基準に織り込む |

## Stage 6: Sharpe比だけに頼らない統計評価

`stats_eval.py` に実装。合成データで**本物のエッジとノイズの区別**を検証済み。

| 指標 | 問い | 実装関数 |
|---|---|---|
| Probabilistic Sharpe Ratio | Sharpeが基準を真に上回る確率（歪度・尖度補正） | `probabilistic_sharpe_ratio` |
| **Deflated Sharpe Ratio** | 何回試したかを踏まえてもSharpeは有意か | `deflated_sharpe_ratio` |
| **PBO** | イン最良がアウトで中央値以下に落ちる確率（CSCV法） | `pbo_cscv` |
| **White's Reality Check** | 最良戦略の優位はデータスヌーピングの産物か | `whites_reality_check` |

検証（合成データ）:

| | 本物のエッジ | 全ノイズ（罠） |
|---|---|---|
| DSR | 1.000（高＝本物） | 0.34（低＝偽物） |
| PBO | 0.00（過学習でない） | 0.41（過学習） |
| Reality Check p | 0.001（優位は本物） | 0.50（優位なし） |

実データ（日経225・SMA63通り総当たり）の`demo.py`結果:
見かけの最良Sharpe0.91も、**PBO=0.54（過学習の方が濃厚）・DSR=0.88**で
「データスヌーピングの産物の疑いが濃い」と判定される ―― これが道具の使いどころ。

## ベクトル型 vs イベント駆動型エンジン（`engines.py`）

| | ベクトル型 | イベント駆動型 |
|---|---|---|
| 計算 | 全時点を配列で一括 | 1本ずつ状態を更新 |
| 速度 | 速い | 遅い |
| 先読み | 混入しやすい（要注意） | 構造的に混入しにくい |
| 経路依存の執行（ストップ・部分約定・遅延） | 表現しづらい | 自然に表現できる |
| 用途 | 大量スクリーニング・ファクター研究 | 執行の精密検証・本番前の最終確認 |

実測（日経225・SMA20/60・2年）: 両者の最終エクイティは一致（差0.0001）。
一方 **ATRストップを足すとイベント駆動型だけ別結果**（1.520→1.502）＝
経路依存はベクトル型では表現しづらい、という設計上の要点を実証。

> 実務のワークフロー: まずベクトル型で広く探索 → 有望なものだけ
> イベント駆動型で執行込みに精査 → ペーパートレード（Stage7）。

## Stage 4: コストモデルの考え方

`engines.py` は手数料＋スリッページを bps で引く簡易版。実務ではさらに:

- **スプレッド**: 気配の内側で約定できない分
- **市場インパクト**: 自分の注文が価格を動かす（大口ほど支配的）
- **約定確率**: 指値が約定しないリスク（成行なら価格悪化）

リテール現実値の目安は片道 **15〜25bps**（`../rakuten-backtest/LEARNING.md`）。
コストを楽観すると、紙上の勝ち戦略が実運用で消える最大の原因になる。

## 限界・注意

- **教育目的の最小実装**。プロダクションの厳密さ（点in-timeデータ、
  正確な約定モデル、税・借株の細目）は持たない
- 生存者バイアスは無料データの制約で未対応（限界として明示）
- DSR/PBO/Reality Check は強力だが万能ではない。**複数の道具で多角的に**見る
- 発注機能なし。実運用は別の正規手段（例: 楽天証券RSS）で行う
